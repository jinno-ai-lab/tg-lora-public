## 1. エグゼクティブ・サマリー：最大のギャップ

現在最大のギャップは、**データセットの過学習問題に伴う、過去の実験データの完全な廃棄（Deprecated扱い）**です。
旧1Kサンプルデータセットでは9Bモデルに対して極端な過学習が生じていたため、過去のすべてのCUDA/PyTorch実験結果（シード42/43/44のウォールクロック短縮、G3タスク精度保持など）は**「Deprecated（歴史的参照用）」**に分類され、論文の主たる主張（一次証拠）からは完全に除外されました。

この結果、以下のギャップが生じています。
1. **有効な実験数値が一時的にゼロ化**: 論文ドラフト（[docs/paper/manuscript.md](paper/manuscript.md)）の主要な実験結果テーブルは枠組みのみが準備され、数値はすべて `TBD`（プレースホルダー）となっています。
2. **論文主張レベル（C1 vs C2）の未確定**: 新5Kデータでの **Gate G2（メモリフロンティア分離）** スイープが未実行であるため、論文のゴールを **C1（効率化システム論文）** に留めるか、**C2（革命的メモリ削減論文）** に引き上げるかが確定していません。

## 3. 主張と証拠のギャップ分析

論文の主張を実証するための具体的なギャップと対策です。

### ギャップ 1: 新5Kデータでの基本評価の不在 (M2 / M3)
- **現状**: 過学習のない正しい 5K データでのウォールクロック短縮率、Loss減少率、VRAM削減率がありません。
- **対策**: 現在バックグラウンドで実行中の `task-213`（`make paper-memory-one-shot`）の完了によって、一撃で 3-seed の数値が埋まります。

### ギャップ 2: 独立した評価データでの品質検証 (M4 / G3)
- **現状**: 旧1K実験では validation セットを downstream 評価に使っており、学術的なクリーンさに欠けていました。
- **対策**: M3 で確定した最良チェックポイントに対し、独立した `test.jsonl` スプリットを用いて `make paper-memory-external-eval` を実行します。

### ギャップ 3: メモリフロンティア分離の証明 (M5 / G2)
- **現状**: **完了 (G2 PASS)**。5Kデータで `MAX_SEQ_LEN=1536/2048` で Baseline OOM / TG-LoRA 成功の分離を確認。30.8% VRAM削減、4.6 GB freed、3-seed confirmed。

### ギャップ 4: Mac (MLX) の結果統合
- **現状**: Mac環境で生成される MLX の数値が未マージです。
- **対策**: Mac側での学習完了後、ディレクトリを rsync 等で `runs/` 以下にマージし、比較レポートを自動生成します。

## 4. 現プランにおける課題と瑕疵の自己レビュー（評価）

現プラン（`master_plan.md`）はデータセットの課題（過学習）を的確に修正しており、極めて論理的ですが、以下の**実行効率における現実的な課題**があります。

### 課題 ①: 総実行時間の長さ（直列実行の限界）
- **詳細**: プラン上の Milestone 2 から 7 はすべて直列で累積すると **100〜200時間以上** を想定しています。
- **対策 (最高率化)**: 論文の claim level (C1 vs C2) を最短で確定させるため、M3 (G1 3-seed) が完了した時点で、M6 (アブレーション+因果) に進む前に、**M5 (G2 メモリフロンティア) を最優先で実行**します。M5 は完了済み (G2 PASS)。アブレーションは3条件分離実験で最小限に抑制済み。

### 課題 ②: 別エージェント（Mac）側でのコード同期
- **詳細**: 我々は Linux 側で MLX のメトリクス保存処理を [scripts/train_mlx_lora_fixed.py](file:///home/jinno/tg-lora/scripts/train_mlx_lora_fixed.py) に追加しましたが、Mac 側を動かしている別エージェントが `git pull` を実行してこの最新コードを適用していない場合、MLX 側の実行で再び「ロスログ記録なし」の課題が再発します。
- **対策**: 別エージェントに対して、実行前に必ず Git から最新の変更を pull して `train_mlx_lora_fixed.py` の修正を有効化することを促し、同期を確認します。
