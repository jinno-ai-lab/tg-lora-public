# Tangent-Gradient LoRA (TG-LoRA) フルフロー全体設計仕様書

本ドキュメントは、勾配速度ベクトルの外挿によるLoRA学習効率化手法である **TG-LoRA (Tangent-Gradient LoRA)**、およびそのコアアルゴリズムである **Prior-based Subspace Learning (M9)** の全体設計と最適化ロジックについて詳述する。

> **本ドキュメントの記述規約** 本仕様は設計意図（Design Intent）と実装実態（Implementation Fact）を区別して記述する。両者が乖離している箇所、およびコードで未確認の箇所は **[UNVERIFIED]** または **[乖離あり]** と明示する。これは2026-06-06時点の調査で、設計通りに実装されていない複数の箇所が判明したことを受けた措置である。

---

## 1. 全体コンセプトとアルゴリズム俯瞰

TG-LoRAは、学習中のLoRA重み変化の「速度（velocity）」を追跡し、局所線形性の成り立つ領域において数式的な「外挿（extrapolation）」を行うことで、オプティマイザによる実 backward パスをスキップ（投機）し、学習時間を短縮するアルゴリズムである。

全体の学習ループは、**【ウォームアップ相】** を経たのち、**【A. 手堅い局所学習（Pilot Step）】** と **【B. 大胆なスキップ（Extrapolation Step）】** の交代サイクルで構成される。ウォームアップ相と本番相の二相化は、序盤の勾配方向が不安定（cos ≈ 0.43〜0.68）な期間に外挿を行わず、通常のLoRA学習で土台を固めるための設計である（詳細は §4.3）。

## 2. Prior-based Subspace Learning (M9)

従来のTG-LoRAでは、固定方向ベクトル v0​ の上で手探りのスケール探索を行っていたため、微小サンプルロスによるスケール誤差が蓄積し、効率が頭打ちになっていた。M9（Milestone 9）設計では、蓄積された軌跡から方向とスケールに関する **Prior（事前分布）** を推定し、低次元の係数空間で有限差分勾配法を用いて最適化を行う。

### 2.1 部分空間（Subspace）の再構築

trainable な LoRA パラメータの要素数を D（非常に高次元）とする。過去の軌跡差分ヒストリー H={ΔWt−d​,…,ΔWt​} から、以下の 3 つの直交基底ベクトル（部分空間）を構成する。

1. **大局的な支配方向（Prior Direction） v0​**: ヒストリー全体の平均方向ベクトル。

   v0​=∥Mean(H)∥Mean(H)​

2. **主成分直交ベクトル u1​,u2​**: 中心化したヒストリーに対して低ランク主成分分析（PCA）を適用し、抽出した第1・第2主成分（pc1​,pc2​）を、グラム・シュミット（Gram-Schmidt）法によって v0​ に対して直交化する。

   Subspace=span(v0​,u1​,u2​)(dim=3)

`w_traj` は直近 `N_initial`（=5）サイクルの delta の median norm として算出される（`extrapolator.py:927-928`）。**[UNVERIFIED]** `w_traj` の実値は現状 `run_metrics.jsonl` に記録されておらず、その大きさと推移は次回 run のログ強化で取得予定。なお現状の lr 崩落（§4.4）により `w_traj` が極小化している疑いがあり、これが下流のスケール問題の上流原因と推定される。

### 2.2 有限差分（Finite Difference）による低次元フィット

部分空間内の座標係数を θ=(α,β1​,β2​)T とする。現在の重み Wt​ に対する外挿重み Wextrap​ を以下のように無次元パラメータ化する。

Wextrap​(θ)=Wt​+N⋅α⋅wtraj​⋅v0​+β1​⋅u1​+β2​⋅u2​

ここで wtraj​ は軌跡ノルムの中央値（無次元化のためのスケール因子）、N は投機ステップ数である。

フィット用ミニバッチ上でのロス関数を Lfit​ とするとき、θ の最適値を以下の有限差分勾配法（ステップ数 S=1, 学習率 ηfit​=0.5）で探索する。

∂α∂Lfit​​≈ϵLfit​(α+ϵ,β1​,β2​)−Lfit​(α,β1​,β2​)​

θ(s+1)=θ(s)−ηfit​⋅∇θ​Lfit​(θ(s))

**[実装実態・確認2]** 最適化変数は α,β1​,β2​ の3係数のみであり、投機ステップ数 N は定数として扱われフィット時に動的調整されない。初期値は α=1.0,β1​=0.0,β2​=0.0、係数に上下限クランプは設けられていない（`extrapolator.py:989-1012`）。フィット用ミニバッチはバッチサイズ1（`m9_batch=1`）である。

**[乖離あり・確認2]** 当初の設計意図は「低次元方向上で進み幅をデータにフィットして決定する（モンテカルロ的探索/有限差分による係数最適化）」ことであったが、進み幅の主要因である N⋅wtraj​ は固定値であり、係数 α の微調整のみがフィットの対象になっている。さらに bfloat16 精度下では FD の perturbation（∼wtraj​×2×10−5）がノイズ域に入り、α が不安定化している疑いがある。バッチサイズ1のFD勾配は分散が大きく、これが大きな N での崩壊（後述）の主因と分析されている。fp32化・バッチ拡大・クリッピング・L2 等の対策は lr 崩落（§4.4）修正後の実測を見てから個別判断する方針であり、現時点では未実装。

- **有限差分コストの会計**: 各パラメータの方向微分（偏微分）を有限差分で求めるため、1ステップあたり dim+1=4 回の Forward 計算が発生する。初期ロス・最終ロスの計算を含め、M9フィット全体で **合計 6 回の Forward（実 backward は 0 回）** が固定コストとして消費される。

### 2.3 N スケーリングの実装実態 **[乖離あり・確認4]**

設計上は `selected_N` が v0​ に乗算され、`raw_norm` は N に比例して増大するはずである。しかし実測では N=5 の raw_norm 平均が 2.217、N=10 の平均が 2.049 と、N が外挿距離を支配していない。これは §2.2 の FD フィットの不安定さ、および §4.4 の lr 崩落による `w_traj` 極小化が複合した結果と推定される。N スケーリング自体のコードロジックは設計通りだが、上流要因により効果が打ち消されている可能性が高い。

## 3. 効率会計（Exact Cost Accounting）

論文の主たる主張である「効率（コスト削減）」を保証するため、TG-LoRAの無駄を含めた全消費コストを実 backward パス数に換算して厳密に計上する。

### 3.1 1サイクルあたりの消費コスト内訳

| 状態 (Flag)                | Pilot BP コスト | 外挿フィットコスト | 答え合わせコスト | 計上される実BP数 | 削減された等価BP数 |
| -------------------------- | --------------- | ------------------ | ---------------- | ---------------- | ------------------ |
| **通常フォールバック (B)** | K×accum         | 0                  | 0                | K×accum          | 0                  |
| **外挿承認 (Y)**           | K×accum         | 6 forwards         | 32 forwards      | K×accum          | N×accum            |
| **外挿拒否 (N)**           | K×accum         | 6 forwards         | 32 forwards      | K×accum          | 0                  |

- **実 backward 数への換算方針**: 外挿・答え合わせ自体はオプティマイザの backward を行わないため、実 backward の消費数は常に **`K * grad_accumulation`** である。たとえ外挿が拒否（Flag=N）されて pilot 状態にロールバックされたサイクルであっても、そのサイクルで実際に行われた pilot 更新の backward コスト（無駄になった実コスト）は実BP数（`full_backward_passes`）に全額計上される。

### 3.2 削減率（`reduction_rate`）の数学的定義

Baseline と TG-LoRA を同一のデータ消化量（等価ステップ数）で比較したときの削減割合を以下のように定義する。

reduction_rate=1.0−Baselineの等価BPTG-LoRAの実BP​=1.0−full_backward_passes+speculative_equivalent_backward_passesfull_backward_passes​

- 外挿が承認（Flag=Y）されたサイクルにおいてのみ、分母の Baseline 等価BPに対して `N * grad_accumulation` が上乗せされ、削減率が上昇する。
- 通常フォールバック（B）や拒否（N）のサイクルでは、実際に行った実コストのみが蓄積し、削減率は相対的に低下する。

### 3.3 現状の効率実測と限界 **[要注意]**

直近の二相化 run（cycle 41/120 で停止）における本番期の accept 率は 12.5%（16サイクル中2件）にとどまり、accept された外挿も lr 崩落の影響で実質的な loss 改善はほぼゼロであった。現状データから「accept パターンをそのまま賢く再現する」最適化を施しても、計算量削減の見込みは約2割程度と推定される。論文の効率主張を成立させるには、§4.4 の lr 崩落修正による accept 率と外挿実効性の回復が前提となる。

## 4. 投機判定ゲートと安定性制御

見込みのない状況での無駄な外挿の試行（Forwardコストの浪費）を防ぐため、コサイン類似度に基づく厳密な制限ゲートを敷く。

### 4.1 コサイン類似度制限（Cosine Consistency Gating）

短期的な勾配方向と長期的な速度ベクトルの類似度 `consistency` に基づき、投機ステップ数 N の最大候補を制限する。

```text
Copyc >= 0.90  =>  最大 N = 20 投機可能
c >= 0.80  =>  最大 N = 10 投機可能
c >= 0.75  =>  最大 N = 5  投機可能
c >= 0.70  =>  最大 N = 3  投機可能
c <  0.70  =>  投機不許可 (N = 0) => 通常LoRA (Flag=B) へ直ちにスキップ
```

類似度が最低ラインの **`0.70`** に満たない場合、外挿の適用そのものを開始せず、M9フィット（6回 forward）や答え合わせ評価（32回 forward）もスキップして、通常LoRAとして安全に更新を行う。

**[乖離あり・確認1] ゲートに用いる類似度の比較対象**: 設計意図では、ゲートに用いる類似度は「M9 delta と velocity の cosine similarity（これから飛ぶ方向と過去蓄積方向の整合）」であるべきだが、実装では `raw_delta_cos_sim = velocity.cosine_similarity(dW)`（`train_tg_lora.py:1787`）として計算されており、ここでの `dW` は **pilot delta（WK​−W0​）** であって M9 delta ではない。すなわち設計が意図した「M9 delta と velocity の cosine」はコード上どこにも計算されていない。なお実測では `raw_delta_cosine_sim` は全サイクルで 0.016〜0.056（あるいは別系列で 0.43〜0.68）と小さく、accept/reject の区別に寄与していない。本項目は metric の意味のズレとして記録するにとどめ、ゲートの比較対象変更は lr 崩落修正後に判断する（現状は pilot delta vs velocity のまま据え置き）。

### 4.2 安定性ガード（Linearity Guard - アブレーション用）

もし Linearity Guard を明示的に有効化した際、以下の異常検知によって能動的フォールバックが行われる。本番実験ではデフォルト無効（`linearity_guard_enabled=false`）。

1. **Trigger A (Velocity Anomaly)**: 速度ベクトルのノルム変化が過去の履歴から 3σ を超えて急増した場合（warmup: 5サイクル以降で判定）。

2. **Trigger B (Unstable Pilot)**: 直前のクイック評価（32件）の最良ロスと現在の `pilot_loss` を比較し、許容マージン（`1.0%`）を超えて悪化した場合。

   Lpilot​>Llast_quick_valid​×(1.0+margin)

3. **Trigger C (Low Acceptance)**: 過去のサイクルでの承認率が閾値を下回った場合（本番実験では自己抑制の罠を避けるためデフォルト無効 `min_acceptance_rate = 0.0`）。

### 4.3 二相化（ウォームアップ解放ゲート）

序盤は勾配方向が不安定で cos が 0.43〜0.68 に集中し（cycle 4〜21 で18サイクル連続観測）、この期間の外挿は崩壊リスクが高い。これは通常のLoRA学習で安定化のため学習率をスモールスタートさせるノウハウと同種の問題であり、序盤は外挿を行わず通常学習で土台を固めるべきと判断した。

- **第1相（ウォームアップ相）**: pilot ステップのみで通常のLoRA学習を回し、外挿（N選択・M9フィット・accept/reject）を一切行わない。
- **解放条件**: `consistency` が `warmup_release_cos = 0.75` を **連続3回**（`warmup_release_count = 3`）超えた時点で解放フラグを True にする。一度解放したら戻さない。
- **第2相（本番相）**: 解放後に初めて外挿を実行。`reduction_rate`・`accept_rate` 等の評価指標は本番相のサイクルからのみ算出し、ウォームアップ相の指標は性能評価に含めない。

直近 run では cycle 26（cos=0.778）で解放され、本番相16サイクル中2件 accept（accept 率 12.5%）であった。

### 4.4 ウォームアップ中の学習率保持 **[修正方針確定・実装予定]**

**[乖離あり・確認5]** 現状の実装では、ウォームアップ相であっても lr 適応の3経路（`adapt_to_convergence`, `adapt_to_acceleration`, `penalize`）が毎サイクル lr を減衰させる一方、回復経路（reward）は外挿が行われないため遮断される。この非対称な減衰により、本番相に入る前（cycle 16〜17 付近）に lr が初期値 5e-4 から lr_min（≈1e-5）まで崩落していた。結果として本番相では `w_traj` が極小化し、N スケーリングが無効化（§2.3）、accept しても実質前進ゼロ（§3.3）という下流症状が連鎖していた。

- **修正方針（実装予定）**: ウォームアップ解放フラグが False の間は lr を **初期値 5e-4 で固定保持**し、外挿系の lr 適応3経路を本番相入りまで全てバイパスする。本番相の最初のサイクルから従来の lr 適応を有効化する。lr_min の値自体は変更しない（崩落の底を上げるのではなく、減衰経路を止めることで対処する）。
- この修正は extrapolator.py の外挿式・係数フィット・N スケーリングには一切手を入れず、lr 制御の前段に状態ゲートを追加する範囲にとどめる。lr 崩落が上流の単一原因であるという仮説（N非支配・accept無効はその下流結果）を、修正後の `w_traj` 回復と N スケーリング復活の実測で検証する。

## 5. 品質保証と評価設計

### 5.1 二階層検証（Double-Tier Validation）

- **第1階層（クイック評価 - 32サンプル）**: 外挿後の accept/reject 判定および Linearity Guard の基準には、学習ミニバッチおよびフル評価セットとは独立した `data/valid_quick.jsonl` から抽出した **32サンプル** を使用する（`quick_eval_examples=32`, `accept_eval_examples=32`）。`valid_quick.jsonl`（32件）は `valid_full.jsonl`（493件）と同一母集団（Capybara/Dolly 由来）だが、互いにディスジョイント（重複なし）なデータ集合である。
- **第2階層（フル評価 - 493サンプル）**: 10サイクルごと、および等価ターゲットステップ到達時のチェックポイントの保存・評価には、検証スプリット全体（493サンプル）を使用し、厳密な汎化性能を測定する。

**[Facts] クイック評価とフル評価のロス差について**: 32件クイック評価のロスは493件フル評価より一貫して 0.38〜0.41 低く出る（checkpoint-250: quick 1.058 / full 1.441、checkpoint-500: quick 1.019 / full 1.424 等）。この差は Baseline・TG-LoRA の双方で同程度に観測されるため、TG-LoRA 固有の過適合ではなく、両評価セットの母集団特性の差に起因する正常な現象である。accept/reject の絶対判定にはクイック評価を用いるが、最終的な品質比較はフル評価（493件）で行う。

**[Facts] 評価条件統一の重要性**: 過去に Baseline 1.2722 vs TG-LoRA 1.4659（差 +0.1937）という大きな乖離が報告されたが、これは Baseline 側の `max_seq_len` 不整合による測定アーティファクトであった。`max_seq_len=1024` で統一して再測定したところ、同一条件での真の差は等価ステップ250で +0.0253（フル493件）と約1.7%にとどまる。全評価は `max_seq_len=1024`・`valid_full.jsonl`（493件）・同一評価関数（`src/eval/eval_loss.py`）で統一すること。

### 5.2 同一データ消化軸での品質比較

早期停止の有無やステップ数の表現の違い（Baseline: 1500 steps, TG-LoRA: 500 cycles）を排除し、両者を完全に公平に比較するため、**同一データ消化量（等価ステップ数: 250, 500, ... 1500）におけるチェックポイント同士の validation loss** を直接対応させた評価表を出力する。

直近の Baseline 実測（フル493件）では、等価ステップ 250→1.441、500→1.424、750→1.448、1000→1.457 と、step 500 を底にほぼ横ばい（軽度の過学習傾向）である。

### 5.3 評価リークの解消 **[Facts]**

過去に、accept/reject の答え合わせ評価がフィット用バッチ（m9_batch）上で行われていたことに起因する評価リーク（差 +0.248 相当の楽観バイアス）が確認され、解消済みである。現在の答え合わせ評価は、フィットに用いないディスジョイントな `valid_quick.jsonl`（32件）で行われ、フィットへの情報漏洩はない。

## 6. 実装構成

コアアルゴリズムは、以下のソースファイル群で構成されている。

1. `src/tg_lora/velocity.py` 勾配速度ベクトル、コサイン類似度（`predicted_consistency`）、および `choose_N` による投機ステップ数決定。`cosine_similarity(dW)` は `velocity.py:308` で計算されるが、現状 `dW` は pilot delta である（§4.1 の乖離参照）。
2. `src/tg_lora/extrapolator.py` PCAとグラム・シュミット直交化による部分空間再構築（`L927-928` 付近で `w_traj` 算出）、および有限差分勾配フィット（`subspace_m9_fit_step`, `L989-1012`）。
3. `src/tg_lora/cycle_state.py` 実 backward パス数・等価ステップ数の管理、および `reduction_rate` 計算。
4. `src/training/train_tg_lora.py` メイン学習ループ。ウォームアップ解放ゲート（§4.3、実装予定）、ウォームアップ中の lr 固定保持（§4.4、実装予定）、無駄な投機の早期バイパス（`proposal_N <= 0` 時の Baseline フォールバック）、独立バリデーションセットを用いた accept/reject 判定、ターゲットステップ到達時の自動 eval/checkpoint 保存（`_check_and_save_linearity_budget_checkpoint`）、`raw_delta_cos_sim` 計算（`L1787`）。
