# 04_acceptance_and_validation.md — TG-LoRAオンライン判定と通常の検証ループの分離

## 1. 事実 (Facts)

### 1.1 TG-LoRA固有のオンライン判定（32サンプル）

*   **概要**: 
    外挿を適用した後のパラメータが崩壊（発散）していないかを検知し、差し戻す（ロールバックする）ための **TG-LoRA独自の制御機構（安全弁）** です。通常のLoRA/QLoRA学習には存在しない機構です。
*   **判定に使用される loss**:
    外挿適用後の検証ロス `loss_after` は、フィットに使用していない独立した検証バッチ（`valid_quick_loader`、32サンプル）を用いて Forward 計算から得られた損失値です。
    *   *コード引用*: [train_tg_lora.py:L2750-2790](file:///home/jinno/tg-lora/src/training/train_tg_lora.py#L2750-L2790)
*   **判定式の実装**:
    外挿前のパイロットロス（`loss_pilot`。これも同一の `valid_quick_loader` で測定されたもの）と比較し、許容閾値 `rollback_tolerance` の範囲内に収まっているかを判定します。
    *   *コード引用*: [train_tg_lora.py:L2800-2806](file:///home/jinno/tg-lora/src/training/train_tg_lora.py#L2800-L2806)
    *   *判定閾値*: `loss_after <= loss_pilot + rollback_tolerance` （デフォルト: 0.005 = 0.5%悪化まで許容）。

### 1.2 通常の学習プロセスとしての検証（493サンプル）

*   **概要**:
    学習全体が正しく汎化しているかを評価するための、**通常のディープラーニングにおける標準的な検証ループ** です。TG-LoRAのアルゴリズム制御やロールバックの判断には一切関与しません。
*   **目的**:
    10サイクルごと、および等価ターゲットステップに達した際の最良チェックポイント（ベストモデル）の保存判定、および Baseline との同一ステップ予測精度の突き合わせ評価（公平な性能比較）のために使用されます。
    *   *該当コード*: [train_tg_lora.py:L506](file:///home/jinno/tg-lora/src/training/train_tg_lora.py#L506), [train_tg_lora.py:L3083](file:///home/jinno/tg-lora/src/training/train_tg_lora.py#L3083)
*   **判定用データ**:
    検証セット全体（`valid_full_loader`、全493サンプル）を使用します。
*   **判定閾値**:
    *   ベストモデル判定: `full_loss < prev_best` (1ピコでも改善していれば保存)。
    *   早期終了判定: `stale_cycles >= early_stop_patience` (指定サイクル数連続で改善しなかったら停止)。

### 1.3 設計上の制約と評価の乖離の解消 (Facts)

1.  **外挿幅における $N$ の適用（物理移動とステップ計上の整合）**:
    *   外挿変位 `m9_delta` の数式において、支配方向 $v_0$ に外挿ステップ数 `selected_N` が乗算因子として適用されるよう修正された ([extrapolator.py:L1024](file:///home/jinno/tg-lora/src/tg_lora/extrapolator.py#L1024))。
    *   有限差分フィット（α, β1, β2 の最適化）の内側でもこの $N$ 倍を前提とした最終変位を目的関数で評価・フィットしている。
    *   これにより、物理的なパラメータ移動距離と、削減率に計上される等価ステップが一致するようになった。
2.  **フィットバッチと判定バッチの分離（データリークの解消）**:
    *   有限差分による係数フィットは `m9_batch`（訓練セットの1ミニバッチ）を用いて行い、最終的な accept/reject 判定（`loss_after`）はフィットに使用していない独立した検証セットである `valid_quick_loader` で評価するようオンライン側が修正された。
    *   これにより、フィットデータを用いた自己採点バイアス（誤 accept の原因）が完全に排除された。
3.  **比較基準データの統一（母集団の統一）**:
    *   `loss_pilot` と `loss_after` の両方が同一の `valid_quick_loader` （検証セット）から計算されるようアライメントされた。
    *   これにより、比較判定のためのベースラインと外挿後の性能が同一の評価データソースで測定されるようになり、評価の乖離が解消された。

## 2. 設計意図 (Rationale)

### 2.1 なぜアルゴリズム制御（Accept/Rollback）に493サンプルではなく32サンプルを使用するのか
*   **計算コスト（学習速度）の維持**:
    TG-LoRAのオンライン制御（Accept/Rollback判定）は、数ステップ（1サイクル）ごとという非常に高い頻度で実行されます。ここで毎回493サンプルの評価を走らせると、`eval_batch_size: 1` の制約下では1サイクルごとに493回の Forward パスが追加され、検証にかかる時間が本来の学習時間を圧倒し、学習高速化という目的そのものが破綻します。
*   **安全弁としての必要十分性**:
    32サンプルの評価値は測定ノイズ（オフライン交差検証レポート [subspace_multi_fold_report.md](file:///home/jinno/tg-lora/runs/offline_tg_w_validation_multi_batch/subspace_multi_fold_report.md) における32件スライス間の実測標準偏差 $\sigma_{32} \approx 0.49\%$ に基づく）を含みますが、外挿が限界を超えて失敗（崩壊）した際のロスの悪化幅（実測レポート [subspace_validation_report.md](file:///home/jinno/tg-lora/runs/offline_tg_w_validation_multi_batch/subspace_validation_report.md) における限界突破時の実測ロス上昇率 約 $+8.8\%$ 〜 $+94.1\%$ など）は極めて大きくなります。したがって、ノイズレベルを遥かに凌駕するシグナル（実測に基づくS/N比は $18\sigma$ 以上）を捉えてロールバックをトリガーする用途（安全弁）としては、32サンプルで統計的に十分機能します。

### 2.2 32サンプルの評価ノイズに対するロバスト性の担保
*   **許容閾値の導入**:
    32サンプルの測定ノイズにより、本来は学習が進んでいるにもかかわらず、たまたま $0.1\%$ 程度ロスが悪化して見えることがあります。これを「1ピコでも悪化したら却下」という厳格な閾値で判定すると、ノイズによる誤却下（False Positiveによる不要なロールバック）が頻発して学習が失速します。そのため、実測された評価ノイズの標準偏差（約0.49%）とほぼ同等の許容バッファ `rollback_tolerance = 0.005` (0.5%) を設けることで、統計的に誤判定を回避しています。
*   **Prior（部分空間制約）による過適合の抑制（仕組みによる担保）**:
    32サンプルという小さなデータのみで最適化（M9フィット）や判定を行うと、通常は容易に「その32サンプルだけに過適合する（疑似相関）」現象が発生します。しかし M9 設計では、そもそも探索する重みの更新空間そのものが、過去の健全な軌跡履歴から抽出した Prior（大局的支配方向 $v_0$ および主成分直交ベクトル $u_1, u_2$）によって構成された低次元（3次元）部分空間に制限されています。高次元の全パラメータ空間をランダムに動くわけではないため、探索空間自体が強い正則化作用を持ち、小サンプルによる評価であっても最適化の方向性が崩れることはなく、実用上十分な汎化整合性が保たれています。

## 3. 通常の学習との違いのまとめ

| 項目 | 通常のLoRA / QLoRA | TG-LoRA |
| :--- | :--- | :--- |
| **毎サイクルの検証** | なし (ステップごとにロスを測定するのみ) | あり (**32サンプルによるオンラインの accept/reject 判定**) |
| **パラメータの巻き戻し** | なし | あり (外挿失敗と判定された場合に直前の状態へ**ロールバック**) |
| **大局的な検証 (493件)** | 定期実行 (進捗監視・ベスト保存・早期終了) | 定期実行 (進捗監視・ベスト保存・早期終了。**通常学習と同一の処理**) |
| **データリーク対策** | 不要 | 必須 (外挿のフィットは `m9_batch` で行い、合否判定は `valid_quick_loader` に分離) |
