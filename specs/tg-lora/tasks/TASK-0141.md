## タスク概要

TASK-0140 は Guard 実験事後解析器（`scripts/analyze_dynfreeze_experiment.py`）の §5.2 品質ゲートが pilot/split 境界 proxy 損失（`loss_valid`）を設計が要求する full-eval 損失（`valid_full`）の代わりに読んでいた欠陥を、**pluggable-receiver パターン**で閉じた: 受口は `loss_valid_full` を優先し、レコードに無ければ `loss_valid` に byte-identical fallback する。直前の変更で `RunMetrics.record_step` に `loss_valid_full` 第一級パラメータを追加し、**キーは `None` のとき書き込まれず**（`extract_loss_and_time(full_eval_only=True)` が `LOSS_VALID_FULL_KEY in r` の存在判定で full-eval cycle を同定するため、無条件 `None` 書き込みは全 cycle を full-eval に見せかけて L\* を破壊する）、`TestFullEvalLossHonestyE2E` が現実の metrics ファイルを通じた完全ラウンドトリップで受口の非不活性を証明した。

**本イテレーションで閉鎖**: trainer（`src/training/train_tg_lora.py`）は full-eval cycle で `full_loss` を計算し `cycle_state.record_full_eval(full_loss)` に渡していたが、**`metrics.record_step(...)` に `loss_valid_full=` として渡していなかった**ため、run_metrics.jsonl は一切の `loss_valid_full` を運ばず、受口は常に pilot proxy に fallback し続けていた —— コードは修正済みに見えるが現実の A/B 比較は汚染されたまま（TASK-0140 docstring・GOAL §7 が警告する「中間指標で結論しない」の静かな違反）。これを本イテレーションで閉じた。

**採用した接線方式**: 当初の「値保存的 hoist」ではなく、**co-locate した専用 record** を採用した。新設した `RunMetrics.record_full_eval_loss(cycle=, full_loss=, total_backward_passes=)` を、既存の GPU-verified `cycle_state.record_full_eval(full_loss)` 呼出と**同一の `is_full_eval_cycle:` ガード直下**に併置する。この co-location により、排出は GPU full-eval 経路が不活性でない限り不活性にならない（誠実性契約が設計した「GPU-verified full-eval 損失のみを信用する」構造と一致）。hoist と違い、純粋加算・既存 record の順序や値への干渉ゼロ・OOM sentinel(`float("inf")`) のような非 GPU 計算経路は誤って接線しない（当該サイトは意図的に未接線）。

## hoist の安全性根拠（値保存性）

各サイトで `eval_loss_detailed(model, valid_full_loader, input_device)` を `record_step` の前に移動しても、**計算結果は同一**である:

1. **model 重みは不変**: `record_step`(L3592) と full-eval(L3802) の間のコード（MLflow logging・pbar・logger・`_check_and_save_linearity_budget_checkpoint`・gold eval）はいずれも model 重みを変更しない。ゆえに hoist 前後で `full_result` は同一。
2. **`record_step` は純粋**: `record_step` は `cycle_state` の `full_backward_passes`・`reduction_rate` のみを読み、`best_loss`/`stale_cycles` には触れない。`record_full_eval` は `self.full_backward_passes`（`record_cycle` が設定・両サイトより前に実行済み）を読むため、順序入れ替えに影響されない。
3. **下流ブロックは元位置維持**: `record_full_eval`・`_evaluate_full_eval_outcome`（`prev_best = cycle_state.best_loss` を full-eval 更新前の値で読む）・checkpoint・early-stop(`break`)・LAWA はすべて元の順序を維持。hoist するのは `eval_loss_detailed` の**計算タイミングのみ**で、副作用の順序は不変。

唯一のリスクは 4000 行ファイル内での機械的編集ミス（インデント・スコープ）であり、`python -m py_compile`・`ruff`・純粋関数 trainer テスト(`test_trainer_loop.py`)で構文面は検証可能だが、**制御フロー検証には GPU 実機 run が必須**（完了条件に明記）。

## 本イテレーションで trainer 接線を入れた根拠

前イテレーションは「trainer 本体ループは GPU 専用・CPU 非実行可能ゆえ、hoist を入れても GPU 実機検証なしでは発火未証明の静かな不活性リスクを抱える」として接線を正式タスクに記録し、受口・排出面・ラウンドトリップ証明に集中する判断をとった。**本イテレーションは feedback の直接指令**「gate の L\* が `loss_valid_full` record から計算されること（かつ trainer が full-eval cycle でそのキーを排出すること）を実 run の metrics ファイル上で end-to-end に表明せよ — さもなくば誠実性契約は一度も活性化せず、コードが修正済みに見えても A/B 比較は汚染されたままである」を受けて供給側を閉じた。

閉じ方は「未証明の静かな不活性」リスクを逆手に取った: (1) hoist ではなく GPU-verified `record_full_eval` と co-locate した専用 record で純粋加算・制御フロー不変、(2) producer 単体テスト・producer→受口 E2E（契約 ACTIVE・L\* が pilot proxy trap を回避）・5 サイト source-inspection 完全性ガードの 3 件で **CPU 上で非不活性を証明**。残るは GPU run で実際の `run_metrics.jsonl` を再生成し DORMANT→ACTIVE 転移を目視する最終一哩のみ（下記「完了条件」の未達項目）。
