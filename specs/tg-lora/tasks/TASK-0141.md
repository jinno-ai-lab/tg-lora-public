## タスク概要

TASK-0140 は Guard 実験事後解析器（`scripts/analyze_dynfreeze_experiment.py`）の §5.2 品質ゲートが pilot/split 境界 proxy 損失（`loss_valid`）を設計が要求する full-eval 損失（`valid_full`）の代わりに読んでいた欠陥を、**pluggable-receiver パターン**で閉じた: 受口は `loss_valid_full` を優先し、レコードに無ければ `loss_valid` に byte-identical fallback する。直前の変更で `RunMetrics.record_step` に `loss_valid_full` 第一級パラメータを追加し、**キーは `None` のとき書き込まれず**（`extract_loss_and_time(full_eval_only=True)` が `LOSS_VALID_FULL_KEY in r` の存在判定で full-eval cycle を同定するため、無条件 `None` 書き込みは全 cycle を full-eval に見せかけて L\* を破壊する）、`TestFullEvalLossHonestyE2E` が現実の metrics ファイルを通じた完全ラウンドトリップで受口の非不活性を証明した。

**残る唯一の未達**: trainer（`src/training/train_tg_lora.py`）は full-eval cycle で `full_loss` を計算し `cycle_state.record_full_eval(full_loss)` に渡しているが、**`metrics.record_step(...)` に `loss_valid_full=` として渡していない**。ゆえに現状の run_metrics.jsonl は一切の `loss_valid_full` を運ばず、受口は常に pilot proxy に fallback し続ける —— コードは修正済みに見えるが現実の A/B 比較は汚染されたままである（TASK-0140 docstring・GOAL §7 が警告する「中間指標で結論しない」の静かな違反）。これが本タスクの閉包対象である。

## hoist の安全性根拠（値保存性）

各サイトで `eval_loss_detailed(model, valid_full_loader, input_device)` を `record_step` の前に移動しても、**計算結果は同一**である:

1. **model 重みは不変**: `record_step`(L3592) と full-eval(L3802) の間のコード（MLflow logging・pbar・logger・`_check_and_save_linearity_budget_checkpoint`・gold eval）はいずれも model 重みを変更しない。ゆえに hoist 前後で `full_result` は同一。
2. **`record_step` は純粋**: `record_step` は `cycle_state` の `full_backward_passes`・`reduction_rate` のみを読み、`best_loss`/`stale_cycles` には触れない。`record_full_eval` は `self.full_backward_passes`（`record_cycle` が設定・両サイトより前に実行済み）を読むため、順序入れ替えに影響されない。
3. **下流ブロックは元位置維持**: `record_full_eval`・`_evaluate_full_eval_outcome`（`prev_best = cycle_state.best_loss` を full-eval 更新前の値で読む）・checkpoint・early-stop(`break`)・LAWA はすべて元の順序を維持。hoist するのは `eval_loss_detailed` の**計算タイミングのみ**で、副作用の順序は不変。

唯一のリスクは 4000 行ファイル内での機械的編集ミス（インデント・スコープ）であり、`python -m py_compile`・`ruff`・純粋関数 trainer テスト(`test_trainer_loop.py`)で構文面は検証可能だが、**制御フロー検証には GPU 実機 run が必須**（完了条件に明記）。

## なぜ本イテレーションで trainer 接線を入れなかったか

trainer 本体ループは GPU 専用・本環境（CPU）では非実行可能であり、5 サイトにわたる control-flow hoist を機械的に実装しても GPU 実機検証なしでは「コードは修正済みに見えるが現実の run で発火するか未証明」という、まさに TASK-0140 が警告した静かな不活性と同形のリスクを抱える。ゆえに本イテレーションは**検証可能な価値**（`record_step` 排出面の第一級化 + 条件付き永続化 + ラウンドトリップ非不活性証明 3 件）に集中し、trainer 接線は all-or-nothing・GPU 検証 gate を伴う本タスクとして正式に記録した。これは feedback が G1/G4 について認めた「証拠を実装するか、gap を具体的 next action + owner で正式記録する」の後者と同じ誠実性原則である。
