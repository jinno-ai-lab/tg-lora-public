## タスク概要

paper-evidence gate 評価器は G0–G4 の5門を持ち、CLAIM ladder C0（G1）→ C1（G1+G3）→ C2（G1+G2+G3 "frontier separation"）を解く。TASK-0144 で導入された 3-state 契約は、**required input が欠損/破損した gate を disproven FAIL ではなく INSUFFICIENT EVIDENCE（unmeasured）に分類**する: GOAL §7「未測定の指標で結論しない」の直接符号化である。G3/G4 はこの契約を履行していた（G3 は a418049 で corrupt input も INSUFFICIENT に拡張済み）。

ところが **G2 だけが履行していなかった**。`_check_g2` は frontier report（G2 の required input・claim C2 の証拠）が与えられなくても `evaluated` を立てたまま `passed=False` を返していた。これは frontier sweep をまだ走らせていない run が、**revolutionary claim C2（frontier separation）を反証された**と読まれることを意味する — §7 鉄則の「中間指標や未測定のままで結論しない」への抵触。しかも silent だった: 全ての支援 infra（`_GATE_REQUIRED_INPUT["G2"]` / `_CLAIM_FOR_GATE["G2"]` / `_known_limitation_for` の G2 分岐 / `consolidate` 側 `_gate_outcome`）は G2 の INSUFFICIENT を正しく扱う用意ができており、producer 1箇所（`_check_g2`）だけが `evaluated=False` を出力していなかった。この silent さは `test_insufficient_does_not_fail_exit_by_default` が「G2.3 needs a frontier report we don't provide」と comment 付きで `--skip-gates G2` していたこと（＝本欠陥の直接の workaround 症状）に現れていた。

本タスクは `_check_g2` を `_check_g3`/`_check_g4` と同一契約に是正する:

1. **3-state frontier tracking**: `frontier_state ∈ {absent, unreadable, loaded}` を導入。`loaded` でのみ既存の G2.3 separation 判定を実行し `passed = all(...)` を返す。`absent`/`unreadable` は `evaluated=False` で bail（INSUFFICIENT）。G2.1/G2.2 は常に計算・報告（透明性）し、detail 文字列は byte-identical。
2. **unreadable reason の surface**: `unreadable` 時は `insufficient_reason` に読み取りエラーを verbatim で載せ、`_known_limitation_for` が gap/root_cause に「unreadable」と surfacing する（「missing」と誤表示しない）。a418049 の G3 版と完全同一。
3. **workaround 除去**: 本 fix で `--skip-gates G2` は不要となり、e2e honesty test 3件が G2 も含めた全門で契約を検証する（workaround は欠陥の症状だった）。

`_check_g2` 以外の生産経路（`_known_limitation_for` / `main()` exit logic / `_GATE_REQUIRED_INPUT` / `_CLAIM_FOR_GATE`）および `consolidate_paper_results.py` は**変更不要** — 既に G2 INSUFFICIENT を正しく扱う用意ができており、producer 修正で初めて活性化する（§6.x pluggable-receiver 誠実性パターン: 受口は正しい・データが届けば即活性化）。
