## 背景（なぜ「run を起動する」ではなく「建築 block を実証し failure-propagation を閉じる」か）

feedback（value-judge）は TASK-0193 を VALUABLE 評価しつつ「robust 化した path で 9B run を実際に起動し §4 TIES を報告せよ」と継続指名する。検証の結果:

1. **両 citable full §4 verdict は既に landed TIES**（`make section4-operator-decision` で `arc_complete=true`・両 leg citable faithful TIES を実測）→ 再起動は既知 TIES の再現（新情報ゼロ）。
2. **9B run は建築的に非実行可能（実証）**: `python -c "import src.training.train_tg_lora"` を実行 → `src/training/train_tg_lora.py` L16 `from src.data.build_seed_dataset import load_dataset` で **`ModuleNotFoundError: No module named 'src.data'`** で即死。`src.data` はこの公開 mirror から意図的に剥奪（`scripts/prepare_data.py` が明記）。ゆえに run 起動は GPU にかかわらず import 段で壁。
3. **この block が failure-propagation gap を顕在化させる**: 本 mirror では**全 `--rerun` が非ゼロ退出（import 失敗）になる**。しかし TASK-0189/0193 の `apply_remediation` は再実行が非ゼロ退出でも結果 status を `"warn"`（非 `"error"`）とし、`main()` は `has_error = any(status=="error")` のみで exit code を決めるため、**再実行失敗が exit code に伝播せず recover.py は exit 0（成功）を返していた**。`--rerun` は自動化（feedback の "report → automatic launch"）のための経路であり、exit code で成功を測る呼び出し元が**失敗した recovery を完走 arm と誤認**する genuine failure mode。

ゆえに run 起動は成立せず、代わりに feedback 命名の高レバレッジ導線（abort→rerun）で未テストの残 sub-path「起動した再実行の失敗を exit code に伝播する」を閉じる。これは「失敗した arm を完走と誤認し partial/不在 deposit を citable verdict として harvest する」上流原因を断つ、§7 誠実性テーマに直結する修正。

## 変更内容

- **`scripts/recover.py`**:
  - 新規 `_rerun_failed(results) -> bool`（`_default_rerun_launcher` の直後）: rerun result（`details["returncode"]` を持つ唯一の result）が非ゼロ退出かを判定。returncode を持たない analyze/sanitize/config result や error result は勘案しない（error は `main()` の `has_error` が別処理）。
  - `main()`: `sys.exit(1 if has_error else 0)` → `sys.exit(1 if (has_error or _rerun_failed(all_results)) else 0)`。再実行失敗をプロセス exit code に伝播。
  - rerun result メッセージ: `f"Corrective re-run launched ({mode}; exit {rc})"` → `f"Corrective re-run {outcome} ({mode}; exit {rc})"`（`outcome="completed" if rc==0 else "FAILED"`）。失敗を人間にも明示（"launched" の曖昧さ排除）。status は従来通り `"warn"`（rc!=0）を維持＝結果 severity と exit code を分離。
- **`tests/test_recover.py`**: 新規 `TestRerunFailureExitPropagation`（4 単体: rc=137 検出 / rc=0 clean / returncode なし clean / error-only clean）+ `TestCorrectiveRerun` に統合テスト 2 件（`--remediate --rerun` で launcher=137 → `main()` 非ゼロ退出 / launcher=0 → exit 0 control）。`_rerun_failed` を import に追加。

## Decision trace

- **run 起動以外の選択肢の排除**: src.data 移植は `prepare_data.py` の "stripped from this public mirror" 注記と矛盾→公開境界を破るため却下。derived-float guard 延長は TASK-0189 EXHAUSTED 宣言→却下。cache/eval hardening は feedback が「diminishing returns・STOP」と明示→却下。新規 metric 周辺 guard は feedback 禁止→却下。
- **なぜ failure-propagation か（他の recover.py 改善でなく）**: TASK-0193 が「doomed resume の起動」を防いだのに対し、本 gap は「起動した再実行が失敗した時の成功誤報」＝同一自動化契約の**未テストの次 sub-path**。feedback が「実際の失敗モードに限定せよ・operator 決定を動かす polish を避けよ」と指示する中、これは「失敗を黙殺しない（§7）」という誠実性の core に直結し、かつ「自動化が recovery 成功を誤認→偽 deposit/verdict harvest」という downstream-decision を反転させる genuine failure mode（polish でない）。
- **なぜ status="warn" を維持し exit code で分離するか**: "warn" は結果の human-readable severity（「要注目」）。exit code は自動化契約の成否。両者を分離することで既存契約（`test_rerun_warns_on_nonzero_exit` が status="warn" を検査）を維持しつつ、自動化への失敗伝播だけを追加＝最小の挙動変更。
- **operator 決定への影響**: 本修正は SHIP/ACCEPT-NULL/PIVOT を直接動かさないが、PIVOT（private-repo で `--rerun` 経由で再実行→deposit→replay）が依拠する robust 起動導線の**信頼性**を高める＝失敗した recovery が偽の完走として harvest される経路を閉じ、citable verdict の真実性を上流で担保する。
