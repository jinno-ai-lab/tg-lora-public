## 背景（なぜこの bug か）

feedback の禁制マップを遵守し、§4 verdict/decision 機構・recover.py・derived-float guard series（EXHAUSTED）・core stats の non-finite「drop」設計（test pin 済み）を回避。TASK-0200/0201 が閉じた「非有限 `grad_norm` 検出」ファミリーの**隣接する別ファミリー**を探し、producer 本体（`train_tg_lora.py`）の full-eval → `best_model` 保存決定に genuine silent-success bug を発見した。これは verdict/prose/eval-adjacent plumbing ではなく、deploy/eval で使われる**保存モデルアーティファクト**と**報告 best loss** を反転させる load-bearing な決定。

## bug（end-to-end 再現可能）

1. `_evaluate_full_eval_outcome`（train_tg_lora.py:539）は `is_new_best = prev_best - full_loss > min_delta` を計算し、docstring（531-538）は「a loss only counts as a new best when it beats `prev_best` by strictly more than `min_delta` (Keras-style)」と**保証**。導入 commit `bce700b`「§5.3 min_delta improvement-margin」が意図を、clobber 修正 commit `73201a4`「stop best_model clobber」が best_model 保存へのかかりを裏付ける。
2. canonical full-eval site（旧 4362）は `is_new_best, should_stop, eval_reason = _evaluate_full_eval_outcome(...)` で `is_new_best` を**束縛するが以降一切読まない**（dead-read・`should_stop`/`eval_reason` のみ使用）。
3. 同行の `cycle_state.record_full_eval(full_loss)`（cycle_state.py:252）は `if self.best_loss - full_loss > self.min_delta:` で正しく gate し `cycle_state.best_loss` を更新。`cycle_state.min_delta = early_stopping_min_delta`（1652 行）で両者は**同一 min_delta・同一 prev_best**。
4. しかし 5 つの `best_model` save site（2732/2869/3037/4362/4613）はすべて `if full_loss < best_full_eval_loss:` raw 比較（min_delta 無視）。

結果: sub-`min_delta` 改善（両 production config `early_stopping_min_delta: 0.01` で発火中）が (a) `best_full_eval_loss` を下げ `best_model` を上書き保存、(b) spurious「New best model」ログ、一方で (c) `cycle_state.record_full_eval` は `cycle_state.best_loss` を更新しない（stale 増）= **保存 best_model と run 公式 best_loss が silent diverge**。`is_new_best` flag が存在し docstring/commit が「new best」を保証するのに action site（best_model 保存）で dead-read = 半配線の未実装バグ（TASK-0200 テンプレと同一形状）。

## fix（Z = policy AND monotonic-best）

5 site すべてを `is_new_best and full_loss < best_full_eval_loss`（4362）/ `was_new_best and full_loss < best_full_eval_loss`（他 4 site）へ gate。`cycle_state.record_full_eval` は new-best かを返すよう変更（`-> bool`・後方互換：戻り値無視 caller は影響なし）。

- **第1 conjunct（`is_new_best`/`was_new_best`）**: §5.3 min_delta 改善マージンを best_model 保存に適用 = 本 fix の核心。
- **第2 conjunct（`full_loss < best_full_eval_loss`）**: 単調ベスト不変量（`best_full_eval_loss` は非増加）。純 new-best gate のみだと、`best_full_eval_loss`（raw・より低い）と `cycle_state.best_loss`（gated・より高い）が diverge した**旧 checkpoint resume** 時、gated best に対する new-best が raw best より悪いモデルで best_model を上書きする回帰が生じる。第2 conjunct はこれを防ぐ（resume-safe）。

**byte-identical 性**: `min_delta=0.0`（既定・config 未設定時）かつ tracker 同期時、`is_new_best` ≡ `full_loss < prev_best` ≡ `full_loss < best_full_eval_loss` なので `A and B` ≡ `B` ≡ 従来 raw 比較。fix は `min_delta>0` 設定時（両 production config）のみ発火。
