## 背景（なぜこれが sibling variant 収穫ではなく、9B run lever への寄与か）

TASK-0189（`4e6f7e9`）は remediation を「resume hint を print するだけ」から「reduced-footprint recovery config で corrective re-run を自動起動する」へ格上げした（feedback の main lever「report → automatic launch」）。しかし導入時の `_build_rerun_command` は **常に** `--resume <run_dir>/training_state.pt` を組み、resume 対象の実在性/整合性を一切判定しなかった。`train_tg_lora.main`（L4825-4832）は `--resume` を `train_tg_lora(cfg, resume_path=...)` へ渡し、L1496-1497 で `load_training_state(Path(resume_path))` を呼ぶ。`load_training_state`（checkpoint.py L841-870）は:

- 対象 absent → `FileNotFoundError`（L844-845）
- torn/truncated → `CheckpointIntegrityError`（L857-869）

いずれも `main()` で catch されず（try/except は `PreflightError` のみ L4826-4829）、起動した re-run プロセスは即座に非ゼロ exit でクラッシュする。**初回チェックポイント保存前に OOM-kill された arm**（OOM は peak memory 確保時の最初の forward/backward で頻発＝最初の `training_state.pt` 保存前に起きやすい）では resume 先が存在せず、feedback の「9B arm killed at step 472→1500」シナリオ（`test_recover.py` `_setup_oom_run` の由来コメント）でもチェックポイント cadence によっては同様。すなわち abort→rerun は「arm を完走させる」目的に対して**最も起きやすいケースで再失敗する**未テスト broken path を抱えていた。

本 TASK はこの broken path を閉じる。**eval-surface/empty-corpus sibling（feedback が閉鎖宣言）でも derived-float guard（TASK-0189 で EXHAUSTED 宣言）でも spine/judge/advisor（feedback 禁止）でもない** — feedback が唯一の high-leverage と繰り返し指名する「9B run を完走させる」tool の堅牢化であり、TASK-0189 の直接後続。

## 実装

### (1) resume 対象の probe — `_default_resume_probe(state_path) -> str`

`"missing"` / `"unusable"` / `"intact"` の3値を返す純粋判定:
- **存在ゲート（常時・torch 不要）**: `state_path.exists()` が False → `"missing"`。これが core fix。OOM-before-first-save の頻出ケースは torch を一切使わずここで捕捉される。
- **torn ゲート（torch-guarded）**: 存在する場合 `load_training_state` を試行。成功→`"intact"`・`CheckpointIntegrityError`→`"unusable"`・その他 load 失敗（partial/foreign blob・schema drift）→`"unusable"`（doomed launch 回避）。torch/checkpoint tooling が import 不可なら存在を信頼して `"intact"`（missing-torch false negative ではなく存在ゲートが決定を駆動）。

### (2) resume 判定 — `_resume_decision(run_dir, *, probe=None) -> (bool, str)`

probe（default=`_default_resume_probe`、注入可能=intact/torn 分岐を GPU/TrainingState 構築なしで unit test 可能＝TASK-0189 の `launcher` 注入 seam と同一パターン）で `<run_dir>/training_state.pt` を判定し、`"intact"`→`(True, "resume from intact ...")`・それ以外→`(False, "... missing/unusable — launching fresh start")`。

### (3) resume-aware `_build_rerun_command(recovery_config, run_dir, *, resume, python=None)`

`resume=True`→`--resume <path>` を付加（従来動作）・`resume=False`→`--resume` を省略し reduced-footprint recovery config で fresh-start（OOM し直さない）。`resume` は**必須キーワード**（呼出側に明示決定を強制＝silent な doomed-resume footgun を排除）。

### (4) `apply_remediation(..., resume_probe=None)` の rerun ブロック

`rerun=True` のとき `_resume_decision(run_dir, probe=resume_probe)` で resume/fresh を決定 → `_build_rerun_command(..., resume=resume)` → launcher 起動。結果メッセージは `(resume; exit N)` / `(fresh start; exit N)` で分岐を明示、`details` に `resume`/`mode`/`resume_reason` を記録。`resume_probe` は `launcher` と同じ注入 seam。
