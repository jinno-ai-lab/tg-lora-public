## 背景（なぜこの bug か）

feedback の禁制マップを遵守し、§4 verdict arc（DONE）・recover.py 軸（飽和）・derived-float guard series（EXHAUSTED）を回避。core stats の非有限処理は「drop, don't surface」の**意図的設計**で test pin 済み（`test_compute_stats_all_nan_returns_zeros` / `test_velocity.py:381-409`）であったため、発散検知の load-bearing な最終防线（`velocity.is_magnitude_anomalous`）は loss/rollback 層で担保されており stats 層の silent-drop は informational・非 load-bearing と判断して却下。

代わりに**テストカバレッジが薄く・offline だが実使用される CLI 診断パス**（`scripts/advise_training.py` → `generate_advice_from_history` → `TrainingAdvisor.evaluate` → `CycleMonitor`）を精査し、genuine silent-divergence bug を発見した。

## bug（end-to-end 再現可能）

1. `scripts/advise_training.py` は `run_metrics.jsonl` を読み、`_extract_cycle_records` が producer 実キー `grad_norm` を抽出（L131・producer は `grad_norm` を書くと L61-62 が明記）。
2. `generate_advice_from_history`（training_advisor.py:599）は **finite train_loss** レコードを `advisor.evaluate(..., grad_norm=record.get("grad_norm"))` へ流す（L650）。
3. `TrainingAdvisor.evaluate` は `cycle_data["grad_norm"]` を設定し monitor へ（L257-259）。
4. `CycleMonitor.update` は `cycle_data.get("grad_norm")` を**読んで捨てる**（dead expression・旧 L66）。
5. `CycleMonitor.detect_divergence` の NaN/Inf チェックは `train_loss`/`valid_loss` のみ。

結果: **有限 train_loss + 非有限 grad_norm** のサイクル（grad-clipping で forward loss は有限のまま、producer 記録の pre-clip grad norm が inf/NaN = 勾配爆発）が発散として検出されず、`CycleMonitor` は `healthy` を返す。CLI は `overall_health=="critical"` で exit 2 する設計（L299）だが、この gap により発散 run が exit 0（healthy）を返す = automation への silent-success。docstring「Detects divergence (loss spikes, NaN gradients)」と `nan_detection=True` 既定が**意図**を、dead read が**半 配線**を裏付ける未実装バグ。

## fix（`src/tg_lora/cycle_monitor.py`）

- `detect_divergence` の NaN/Inf ループに `("grad_norm", current.get("grad_norm"))` を追加（loss 2 entry を先にチェックするため非有限 loss が `metric` ラベルに勝つ優先順位は不変）。
- `update` の dead `cycle_data.get("grad_norm")` を削除し、grad_norm は history から読まれる旨の短い pointer comment に置換。
- `recommend_intervention` は critical severity → "rollback: NaN/Inf detected" + "reduce_lr" を自動生成するため、recommendation path は追加変更不要。
