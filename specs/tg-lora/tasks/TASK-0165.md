## 背景（resume-ledger gate の MATCHING 半分が assembled で未 pin）

resume-ledger gate は**二方向**を持つ:

1. **MISMATCH**（config 変更）→ ledger を無視して全 arm を再実行（silent replay を防ぐ）。
2. **MATCHING**（真の resume — GPU window を跨ぐ再 fire）→ ledger から全 arm を replay し、fresh one-shot run と**同一** verdict / losses / CIs / provenance / evidence_hash を assemble する。

fingerprint gate siblings（lora_r / learning_rate / max_dataset_rows / 13-dim parametrization）は全て **(1) MISMATCH 半分** を pin した:各 dim を変えると arm runner が再呼び出しされ、stale arm は replay されない（`test_*_fingerprint_gate` 群が `call_count == 4` で確認）。`tests/test_freeze_validloss_ci_9b_resume.py` は replay **mechanics** を stub runner で unit test するが、`run_ci_9b` assembled は走らせない。

つまり **(2) MATCHING 半分** — gate の定義そのものであり、multi-window launcher architecture（[TASK-0161](TASK-0161.md)/[TASK-0162](TASK-0162.md)）が各 arm をこの ledger に bank して GPU window を跨いで resume するという headline 成果物が依存する property — は、assembled scale で pin する invariant が存在しなかった。replay された arm の `(valid_loss, provenance)` が assembled join（`surrogate_valid_loss_ci` → `result_to_json` → cost accountant / regime classifier）を通って verdict に至るまでのどこかで腐っても、unit test（`_collect_arms` の mechanics のみ）は green のまま潜在化する（GOAL §7 corrupt-but-green）。
