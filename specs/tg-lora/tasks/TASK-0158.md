## 背景（完全 §4 verdict run が中断で無に帰する問題）

TASK-0157 で citation gate に 4th conjunct（regime 軸）が加わり、`make freeze-validloss-ci-9b-full`（600 train / 1500 step・9 arm = 3 candidate + 3 surrogate + 3 baseline）が**完全 §4 verdict を clearing する唯一の path** として確定した。しかし同 target は **~半日分の 9B GPU**（12GB RTX 3060・`Qwen/Qwen3.5-9B` 4-bit・seq1024・9 arm × 1500 step）を要し、このカードは**併発する private-repo run に日常的に抢占される**（本 iter も 28:39 elapsed の 9B run が GPU を占有中）。問題は `run_ci_9b` が**全 arm を in-memory に集め、deposit を最後にだけ直列化**していたこと — 深い位置（例: 8 arm 完了後の最後の baseline）での中断は**完了分の全 arm を失い**、verdict は「次の free-GPU window に最初の 9 arm から再開」する永遠の未達に陥る。feedback の指した「SINGLE highest-leverage non-edge-case task」は**verdict 取得経路そのものの unblock**（proxy/control の polish でも gate の polish でも 2nd scaled-smoke でもない）であり、本 iter の問題意識は「半日の 9B run を free-GPU window の集積で段階的に着地可能にする」こと。

**GPU block の誠実記録**: 本 iter で real 9B full-budget verdict は取得できなかった（GPU 抢袭中・~半日要する）。しかし完全 verdict run が「1 回の中断で全部失われる」脆さは、**ledger 1 つで構造的に除去できる GPU-free な欠陥**だった。これを閉じたことで、次の free-GPU window は 9 arm を 1 発で走らせる必要がなくなり、window 単位で arm を bank していく形で完全 verdict に近づける — verdict instrument の polish ではなく、verdict 取得 critical-path の unblock。

## 実装（`scripts/run_freeze_validloss_ci_9b.py`）

### 1. opt-in resumable per-arm ledger

`--ledger PATH`（default `None` = byte-identical 従来挙動・既存 5 deposit は全て no-ledger で不変）。`LEDGER_VERSION = 1` + `IncompleteResumeError(RuntimeError)`。ledger は **JSONL**: 1 行目 = header（`run-config fingerprint`）、以降 = 完了 arm 1 行（`{type:"arm", role, index, valid_loss, provenance}`）。fingerprint は run-config の **決定的 hash**（`total_steps` / `warmup_steps` / `depth` / `spacing` / `seq_len` / `train_examples` / `valid_examples` / `model` / `scope_label` / `active_scope` / `dataset` / `use_local_loss` / `base_seed` 全てを含む）で、96-step smoke の ledger が 1500-step full run に誤って reuse されるのを防ぐ。

- **`_arm_specs(...)`** — 4 ロール（candidate / surrogate / control / baseline）× n_* の spec list を構成。`run_ci_9b` 内にあった 4 つの inline comprehension を**意味的に完全一致**で引き出した factory（`test_arm_specs_match_legacy_semantics` が role/order/seed/depth を各 arm で pin）。実行順序は candidate → surrogate → control → baseline（headline A/B が最も早く完了）。
- **`load_ledger(path, fingerprint)`** — `{(role,index): (valid_loss, provenance)}` を返す。header の fingerprint が不一致 → **stale として空**を返す（96-step ledger を 1500-step run で無視）。trailing line が破損（crash 途中の flush）→ **その行を skip**（resume が brick しない）。file なし → `{}`。
- **`append_arm_to_ledger(...)`** — stale/missing なら header を新規書き、match なら append。
- **`_collect_arms(specs, runner, *, ledger_path=None, fingerprint=None)`** → `(collected, n_resumed)`。各 spec について ledger に bank 済みなら**再利用**（runner を呼ばない）、未 bank なら `runner(spec)` を呼んで結果を ledger に append。`ledger_path=None` で **byte-identical 従来挙動**（全 spec 実行・`n_resumed=0`）。
- **`_require_runnable_arms(candidate_losses, surrogate_losses)`** — headline A/B の両 arm が揃っていないと verdict が組めないので `IncompleteResumeError`。`main` はこれを catch して **deposit を書かず exit 3**（誠実な「まだ未完」・同じコマンドを再実行して gap を埋めるよう促す）。
- **`_make_arm_runner(model, ...)`** — per-spec GPU arm runner closure を module-level factory に抽出。`run_ci_9b` の `finally: del model` と closure の free variable が衝突して ruff F821 を誘発していたのを、`model` を factory の**引数**にして静的に解決（factory 内では `del` されない）。

### 2. additive deposit keys

`result_to_json` に `resumed_arm_count`（int・default 0）と `ledger_path`（str|None・default None）を**additive** に追加。既存 5 deposit は no-ledger なので `result_to_json` の legacy dict は `.get` default で安全に直列化される（`test_result_to_json_resume_keys_default_for_ledgerless_result` で pin）。

### 3. Makefile 接続

`FREEZE_9B_FULL_FLAGS` に `--ledger runs/freeze_validloss_ci_9b_full_ledger.jsonl` を追加（`runs/` は gitignored で ledger は commit されない）。他の target（`-9b` / `-generalization` / `-baseline`）は**意図的に ledger なし**（短時間 run で resume 価値 < 複雑度）。

## 受け入れ test（`tests/test_freeze_validloss_ci_9b_resume.py`・+17・GPU-free・mutation 証明）

stub `runner`（GPU model を load しない）で ledger 純論理を pin:

1. **spec-builder 等価性**（`test_arm_specs_match_legacy_semantics`・`test_arm_specs_execution_order_is_candidate_first`）— refactor が legacy comprehension と完全一致（fresh run は従来と同一）。
2. **fingerprint invalidation**（`test_fingerprint_changes_with_total_steps` / `_active_scope` / `_use_local_loss` / `_carries_ledger_version`）— config 変更で stale 判定。
3. **ledger I/O**（`test_ledger_roundtrip` / `test_load_ledger_missing_file_returns_empty` / `_ignores_stale_fingerprint` / `_tolerates_torn_trailing_line`）— roundtrip / 欠損 / stale / torn 耐性。
4. **resume-skip**（`test_collect_arms_fresh_runs_all_when_no_ledger` / `_resume_skips_banked_arms` / `_banks_newly_run_arms_then_second_pass_hits_all` / `_stale_ledger_runs_everything`）— headline 価値: bank 済み arm は再実行せず、missing arm だけ走る（`call_log == [("surrogate", 2)]`・`n_resumed == 5`）。
5. **incomplete-resume guard**（`test_require_runnable_arms_raises_when_either_side_empty` / `_passes_when_both_present`）— A/B 片側欠損で `IncompleteResumeError`。
6. **additive compat**（`test_result_to_json_resume_keys_default_for_ledgerless_result`）— legacy dict の直列化。

**mutation 証明**: resume-skip の skip を壊す（bank 済み arm も実行する）と `test_collect_arms_resume_skips_banked_arms` + `_banks_newly_run_arms_then_second_pass_hits_all` が RED → revert で GREEN。
