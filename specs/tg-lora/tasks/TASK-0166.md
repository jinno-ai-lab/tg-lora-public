## 背景（signal-kill が FATAL 扱いで multi-hour run を stranded にする）

`scripts/launch_freeze_ci_9b_full.py::classify_exit_code` は worker の exit code を RETRY/FATAL に分類する純粋関数。契約は `0/75/3/2/1` の 5 code + それ以外は catch-all `FATAL "unknown"`。従来 `tests/test_launch_freeze_ci_9b_full.py::test_negative_signal_is_fatal` は **returncode `-9`（負）を FATAL に固定** し、コメントに *"A subprocess that never started (returncode -N) is not a tempfail."* と記載していた。

この前提は **Python `subprocess` の POSIX 仕様に関して事実誤り**:
- 負の returncode は *exclusively*「child が signal N で終了した」（`-9` = SIGKILL / `-15` = SIGTERM / `-11` = SIGSEGV）。検証済み（`subprocess.run` で SIGKILL した子 → `-9`）。
- exec 失敗（binary not found 等）は**正**の `127` を返す（負ではない）。
- interpreter binary 欠落は `FileNotFoundError` を**親側で raise** し、returncode は存在しない（= classify は呼ばれない）。

ゆえに returncode `-N` は常に「worker がホストから signal-kill された」を意味し、"never started" ではあり得ない。本 launcher が存在する shared/contended host では、OOM-killer が最大 RSS プロセス = 9B worker（5.5 GiB+）を **SIGKILL** するのが正に定型 failure mode である。この時:
- ledger は bank 済み arm を保持（`append_arm_to_ledger` は `flush()` 単位で durable・mid-arm kill は torn line → `load_ledger` skip → 当該 arm が再実行、整合性保全）。
- 次の free-GPU window で再 fire すれば run が完了する。

にもかかわらず FATAL 分類では launcher が**即座に停止**し、banked arm が transient な host-kill の背後に stranded となり、operator の手動再 run を要求する。これは `4afc5e9`（run-time CUDA OOM を tempfail 化して launcher が窗口を poll し続ける）と**同一 class** の silent contract-break であり、`4afc5e9` は例外経路の OOM を閉じたが signal-kill 経路の death を閉じていなかった残存 gap。

## 実装（`scripts/launch_freeze_ci_9b_full.py`）

`classify_exit_code` に、catch-all `FATAL "unknown"` の**直前**に負 code branch を追加:

```python
if code < 0:
    # Signal-kill: transient host event (OOM-killer / host SIGTERM), NOT a
    # logic error — retry the next window; the --ledger banks progress. The
    # bound (max_attempts / deadline_seconds) prevents infinite spinning on
    # a genuinely-broken worker.
    return Decision(Action.RETRY, "signal", tempfail_sleep)
```

- 負 code → `RETRY` / kind `"signal"` / `tempfail_sleep`（exit 75 と同一 — free-GPU / stable window 待ち）。SIGKILL(`-9`)/SIGTERM(`-15`)/SIGABRT(`-6`)/SIGSEGV(`-11`) 全て RETRY。
- **正**の未知 code（`127` = exec-failure 等）は従来通り `FATAL "unknown"`（host-kill ではなく非 transient なので retry は無限 loop になる）。これが過剰 reach でないことを test で pin。
- retry は `--max-attempts`(default 600) / `--deadline-seconds`(default 12h) で bound されるので、真に壊れた worker は `RETRIES_EXHAUSTED` で表面化し（無限 spin しない）、banked arm は保全される。
- module docstring の exit-code 契約表に `-N worker killed by signal N -> RETRY` 行と rationale を追加。

## test（`tests/test_launch_freeze_ci_9b_full.py`・mutation 証明）

従来の `test_negative_signal_is_fatal`（事実誤りの前提で FATAL を固定）を削除し、以下に置換:

1. `TestClassifyExitCode::test_negative_signal_is_retriable_tempfail` — SIGKILL/SIGTERM/SIGABRT/SIGSEGV（`-9/-15/-6/-11`）各々が `RETRY` / kind `"signal"` / `sleep=120`。POSIX 仕様の根拠を docstring に明記。
2. `TestClassifyExitCode::test_unknown_positive_is_fatal` — 正の未知 code（`127` = exec-failure）は `FATAL "unknown"` のまま（fix の過剰 reach 防止・"never started" は FileNotFoundError で到達不能も明記）。
3. `TestRunLoop::test_signal_kill_then_done` — run-loop level の証明: `[-9, 0]`（kill → 再 fire → 成功）が `Outcome.DONE` / `attempts==2` / `sleeps==[120]` / worker 2 回呼出（kill された worker を再 fire し stranded にしない）。`test_tempfail_then_done` の signal 版。

**mutation 証明**: `if code < 0:` を `if code < -9999999:` にして branch を unreachable にすると (1) と (3) が RED（`-9` が catch-all `FATAL "unknown"` に落ちる）・復元で GREEN。正の未知 code の test (2) は mutation 下でも緑（= branch は signal 専用・過剰 reach でない）。
