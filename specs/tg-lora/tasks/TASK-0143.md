## タスク概要

AI Hub feedback（前イテレーション）の #1 最高効率指示は「§5.2 修正が不活性でないことを証明せよ: gate の L\* が `loss_valid_full` から計算されること、かつ trainer が full-eval cycle でそのキーを排出すること —— さもなくば誠実性契約は発火せず、A/B 比較はコードが修正済に見える間ずっと汚染され続ける」とした。

2693ddf は受口側を閉じた: `TestFullEvalLossHonestyE2E` が現実の metrics ファイルを通じた完全ラウンドトリップで、L\* が `loss_valid_full` から計算されること（受口が休眠しないこと）を証明した。しかし**供給側（trainer 接線）は TASK-0141 として GPU 専用タスクに記録されたまま**である。これは実用上、決定的な帰結を持つ: **今日の全実 run は `loss_valid_full` を 1 件も運ばない**ため、analyzer の受口は常に pilot proxy に byte-identical fallback する。コードは「修正済」に見えるが、現実の全比較は汚染された proxy 上で行われている —— まさに feedback が警告した静かな不活性状態である。

trainer 接線は本環境（CPU・private `src.data` 欠如）では機械的に閉じられない（TASK-0141 の all-or-nothing + GPU 検証 gate）。ゆえに本イテレーションは、**その休眠を所有付きで可視化する**ことで誠実性契約を強化する: analyzer が pilot proxy にフォールバックしたとき、それを黙示の byte-identical fallback としてではなく、構造化された目立つ警告として（stdout + `gate_decision.txt` の両方に）報告し、gap を TASK-0141 に帰属させる。これにより、誤って「修正済」と読まれる汚染比較は二度と黙って出荷されない。DORMANT 診断は fail-conditioned であり（`loss_valid_full` を運ぶ record が ≥1 件あれば ACTIVE に反転し警告は消える）、TASK-0141 が GPU で閉じられた瞬間に自己消去する —— TASK-0142 の `known_limitation` 機構と同じ自己陳腐化防止設計である。

実装は 3 層:
1. **検出層** (`honesty_contract_status`): step record のうち `loss_valid_full` を運ぶ件数を数え、`{state: active|dormant, step_records, full_eval_records}` を返す。状態は件数駆動（0 件 → dormant、≥1 件 → active）であり、coverage 比（M/N）で部分排出 trainer も可視化する。非 step record（footer 等）は分母から除外する。
2. **可視化層** (`format_honesty_contract_line`): dormant → `⚠ Honesty contract DORMANT: 0/N cycles carry loss_valid_full — L* is on the pilot proxy (loss_valid), NOT the design-mandated full-eval loss; the A/B comparison is contaminated until the trainer emits it (TASK-0141, GPU).` / active → `Honesty contract ACTIVE: M/N cycles carry loss_valid_full — L* computed from the full-eval loss.` 決して空行を返さない。
3. **配線層** (`main()` + `write_gate_decision()`): baseline records に対して状態を計算し、stdout に印字するとともに `gate_decision.txt` の L\* 行の直下に永続化。`write_gate_decision` の `honesty_status=` は optional（default `None`）で後方互換。

## 信頼性レベルサマリー

- **総項目数**: 3 層（検出・可視化・配線）+ 6 件 corrupt-input テスト
- 🔵 **青信号**: 検出機構・可視化機構・配線・6 件の corrupt-input E2E テスト（現実ラウンドトリップ上）・後方互換の optional param
- 🟡 **黄信号**: なし（状態は件数駆動で決定的）
- 🔴 **赤信号**: trainer 接線（TASK-0141・GPU）— 本環境では実行不能のため、本タスクはその休眠を可視化する側から閉じた

**品質評価**: 高品質 — feedback #1 の「誠実性契約は発火せず A/B 比較は汚染され続ける」懸念に対し、供給側（TASK-0141）が GPU で閉じられるまでの間、その汚染を所有付きで可視化し続けることで「修正済に見えるが不活性」状態を不可能にした。doc-only ではなく analyzer の振る舞い差分 + corrupt-input テストとして検証可能。TASK-0141 完了時に fail-conditioned 設計で自己消去し、記録が陳腐化しない。

**なぜ本イテレーションで trainer 接線（TASK-0141）を入れなかったか**: 5 サイトにわたる値保存的 hoist を GPU 実機検証なしで入れるのは、TASK-0140/2693ddf が警告した静かな不活性と同形のリスク（コードは直ったように見えるが実 run で発火するか未証明）を抱えるため、TASK-0141 の all-or-nothing + GPU 検証 gate を尊重した。代わりに、CPU で完全検証可能な「休眠の可視化」で誠実性契約を強化し、gap が黙って放置されることを防いだ。
