## タスク概要

AI Hub feedback（前イテレーション）は「G1/G4 は現在 FAIL（`docs/master_plan.md` L123-136）。最高効率の次手は基礎 paper claim を進めること: 証拠/ablation を実装して G1/G4 を PASS に反転させるか、gap を具体 next action + owner で既知制限として正式記録せよ — gate を failing 状態で放置するのではなく」と指示した。

本リポジトリの public mirror は GPU を持たず、private `src.data` パイプラインも欠くため、G1/G4 を PASS に反転させる実機実験（M6 の 3-seed final-eval-only sweep・cache-isolation ablation）は**本環境では実行不能**である。ゆえに本イテレーションは feedback が認めた後者（gap の正式記録）を採る。ただし doc-only ではなく**検証可能な振る舞い差分**として: gate 評価器が FAIL を返すとき、その FAIL が「なぜ失敗し・どの実験が反転させ・誰が閉じるか」を構造化レコードで自己記述するようにする。これにより FAIL は「回帰かもしれない無言の失敗」から「所有者付きの追跡可能な既知制限」に変わり、gate の再実行ごとに「何の証拠が足りないか」が報告される。

実装は 3 層:
1. **導出層** (`_known_limitation_for`): gate 結果の `passed`/`checks` を読み、PASS なら `None`、FAIL なら失敗した sub-check 名の接頭辞（`G1.1`/`G1.3` = wall-clock、`G1.4` = quality、`G4.1` = warm-speedup、`G4.2` = cache-memory）から gap を組み立てる。静的文字列ではなく check 駆動 —— これが「品質は PASS なのに wall-clock のせいで FAIL」を正しく帰属する鍵。
2. **装飾層** (`@_attach_known_limitation`): `_check_g0`…`_check_g4` を包み、各結果に `known_limitation` を付与。main() の `report_data["gates"] = results` と `_format_report(results)` の両方に自動で流れる。
3. **可視化層** (`_format_report`): FAIL gate の下に `⚠ Known limitation: <gap>` / `Root cause:` / `Next action:` / `Owner: (blocks: <claim>)` を印字。bare FAIL ではなく人間が読める行動可能情報。

- **総項目数**: 3 層（導出・装飾・可視化）+ 2 gate（G1/G4）の閉包基準

- 🔴 **赤信号**: G1/G4 を PASS に反転させる GPU 証拠（M6）— 本環境では実行不能のため next action として記録

**品質評価**: 高品質（記録機構）・要 GPU 検証（claim 反転）— feedback が「証拠を実装するか、gap を next action + owner で正式記録するか」とした後者を採り、かつ doc-only ではなく gate 評価器の振る舞い差分 + corrupt-input テストとして検証可能にした。G1/G4 が M6 で PASS に反転すれば `known_limitation` は fail-conditioned 設計により自動で `None` となり、記録が陳腐化しない。

**なぜ本イテレーションで G1/G4 を PASS にしなかったか**: 両 gate の反転は GPU 実機実験（3-seed final-eval-only sweep / persistent-optimizer cache-isolation ablation）を要し、本 public mirror 環境（GPU なし・private `src.data` 欠如）では実行不能。機械的に閉じられない GPU 証拠を偽造せず、TASK-0141 と同じ誠実性原則で next action + owner + 閉包基則として正式記録した。
