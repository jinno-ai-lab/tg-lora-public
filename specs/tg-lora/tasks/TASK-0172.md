## 背景（replay は cited losses を committed ledger に bind しない）

`scripts/replay_freeze_validloss_ci.py` は任意の committed deposit を GPU-free で再判定する chokepoint である。現状の検査は全て **intra-deposit** に閉じる:

- `replay_samples` が `candidate_losses`/`surrogate_losses` から verdict を再導出し `faithful` で stored `verdict` と照合する ✓
- `evidence_hash` を deposit の evidence fields から再計算し stamped hash と照合する ✓
- `citation_label_stale` が top-level citation boolean を 4 軸 artifact から再導出（`d734327`）✓
- `direction_verdict_stale`/`baseline_verdict_stale` が nested sub-verdict label を per-arm losses から再導出（`371e934`）✓
- `negative_control_flagged`/`full_context_effective` が operator-set boolean を artifact reality から再導出（`9dff092`/`bbf6e68`）✓

しかし **verdict の計算元である per-arm losses 自体を、harvest 時の一次記録（committed ledger `ledger_witness_path` の per-arm `valid_loss`）と照合する検査は一つもない**（`ledger` parse/grep 0）。両 citable full-budget deposit は `*_ledger.jsonl` を同梱し（`4b88ca8`/`1224057`）、その ledger は各 arm の ground-truth `valid_loss` を記録するが、replay はそれを読まない。

ゆえに**内部的に完全に整合した** hand-edited / externally-supplied deposit:

- `evidence_hash` は**編集後** field と一致する（hash を編集後に再 stamp すればよい）
- recorded `verdict` は**編集後** losses と一致する（`replay_samples` が編集後 losses から再算出するので `faithful:True`）
- 全 citation/sub-verdict gate が green のまま

が committed ledger と乖離しても、stale field も note も**存在せず完全 silent** に通る。これは corrupt-but-green §4 verdict を成立させる silent contract-break（GOAL §7）。producer が書いた直後の deposit では到達不能（producer は ledger に arm を bank しつつ losses に同じ値を格納する）だが、replay は**任意の artifact を再判定する chokepoint** であり producer の善意に依存してはならない。

**到達性（実証済 proof of need）**: committed `freeze_validloss_ci_9b_full.json`（recorded **TIES**・`citable_as_full_section4_verdict=True`）の `candidate_losses[0]` を **−0.05** すると、replayed verdict が **SURPASSES** に flip し**citable のまま** = honest null を claimed §4 win に化けさせ、`faithful`/`evidence_hash`/`citation_label_stale`/`*_verdict_stale`/`negative_control`/`full_context` の**全 intra-deposit gate が green のまま** silent に通る。ledger binding のみがこれを捕まえる。

## 設計判断（intra-deposit cross-check の deposit-vs-ledger 延長・artifact-when-present discipline）

- **新 class**: `371e934`/`d734327`/`9dff092`/`bbf6e68` は全て「stored label vs **同一 deposit 内**の artifact」の intra-deposit 検査。本 TASK は「**deposit の losses vs 別 file の committed ledger**」の **deposit-vs-ledger** 検査 = 同 chokepoint（replay gate）・同 §6.2 prose/JSON drift 不能構造だが**異 class**（照合対象が同 deposit 内でなく外部一次記録）。
- **artifact-when-present discipline**: ledger を持つのは両 citable full-budget deposit（7 つの 9B deposit 中 2 つ）のみ。proxy/synthetic/reduced-budget の 5 deposit は `ledger_witness_path` を持たない。ledger がない deposit で cross-check は**skip**（clean を返す）し false-positive しない = `9dff092`/`bbf6e68`/TASK-0171 の arm-未実行 slot skip と同 posture。`_load_committed_ledger` が `None`（path 不在/file 不正含む）なら `_ledger_losses_stale` は `ledger_losses` 空 → 即 `[]` return、`_ledger_witness_stale` は `not records` → `False`。
- **単一 parse・再利用**: `format_replay`（prose）と `replay_to_json`（JSON）の**両方**が `_load_committed_ledger(data, path)` を 1 回呼び、その `records` を `_ledger_losses_stale`/`_ledger_witness_stale` に渡す = §6.2 prose/JSON drift 不能（`371e934`/`d734327` と同じ構造）+ 2 read 間の TOUCA race を排除。
- **canonicalization 一致**: witness hash の再導出は harvest 時（`fire_freeze_ci_9b_full_heterogeneous.sh` + `TestCommittedLedgerWitness._ledger_witness_sha256`）と同一の `json.dumps(records, sort_keys=True, separators=(",",":"))` を使う = committed corpus で `ledger_witness_stale==False`（byte-identical）。

## fix

`scripts/replay_freeze_validloss_ci.py`（`import hashlib` 追加）:

1. `_LEDGER_ROLE_LOSS_KEYS` 定数 — `(role, deposit_losses_key)` の対（candidate/surrogate/control/baseline）。main verdict と direction/baseline sub-CI が描く verdict-critical 4 arm を cover。
2. `_resolve_ledger_path(data, deposit_path)` — `ledger_witness_path` を CWD-relative（`make freeze-replay` の repo-root 起動）+ deposit 自身の dir/祖先（絶対 path deposit 起動）で解決。不在時 `None`。
3. `_load_committed_ledger(data, deposit_path)` — JSONL を parse し record list を返す。path 不在・parse 失敗時 `None`。
4. `_ledger_arm_losses_by_role(records)` — `type=="arm"` record を role+index で sort し per-seed `valid_loss` vector を構築（producer が bank する arm 順序と同一）。
5. `_ledger_losses_stale(data, records)` — 各 role の deposit `<role>_losses` と ledger vector を長さ + 要素wise（`float(a) != float(b)`）で比較し不一致 role 名の list を返す。`ledger_losses` 空（ledger 不在）時は即 `[]`（skip）。empty==empty は match。
6. `_ledger_witness_stale(data, records)` — stamped `ledger_witness_sha256` と `json.dumps(records, sort_keys=True, separators=(",",":"))` の SHA-256 を比較し不一致で `True`。stamp 不在/records 空時 `False`。
7. `format_replay`: `BASELINE_VERDICT_STALE` block の直後に `LEDGER_LOSSES_STALE`/`LEDGER_WITNESS_STALE` note（stale 時のみ・対処＝fresh producer run で re-harvest）を出力。
8. `replay_to_json`: return dict で `ledger_records = _load_committed_ledger(data, path) or []` を 1 回計算し `ledger_losses_stale`/`ledger_witness_stale` の 2 field に再利用（`*_verdict_stale` の直後）。

保証: **ledger と乖離する internally-consistent deposit は `ledger_losses_stale` に role 名 + `LEDGER_LOSSES_STALE` note・witness hash 嘘は `ledger_witness_stale=True` + `LEDGER_WITNESS_STALE` note で fail-loud**。producer が書いた committed corpus では deposit losses == ledger losses かつ witness hash 一致なので stale=False/[]（byte-identical）。gate と prose は同一 helper 経由で drift しない（§6.2）。
