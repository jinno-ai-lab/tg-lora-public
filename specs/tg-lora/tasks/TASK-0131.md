## タスク概要

`ProgressiveFreezeController` は Phase 1 では**単層ゲート**（`_is_frozen` bool・2回目の `apply_freeze` は "already frozen" で拒否）として実装されていた。design §4.1 が定義する「Progressive」Freezing の核心——1回の run 内で X を T1、X-1 を T2、X-2 を T3 …と凍結集合が単調増加していく機構——を駆動する単一オブジェクトが存在しなかった。TASK-0130 の E2E は `freeze_schedule` が予測する多層計画 `{5:1, 4:2, 3:3, ...}` を、**層ごとに別コントローラを生成**するワークアラウンド（`_drive_schedule`、docstring に "the controller is a single-freeze gate" と明記）で実行していた。

本タスクは、単一コントローラが `FreezeSchedule` に駆動されて多層を漸進凍結する native API を追加し、このワークアラウンドを不要にする。単層（Phase 1）挙動は完全保持する。

## 主要実装項目

### Progressive multi-layer API（design §4.1）🔵

**信頼性**: 🔵 *design §4.1・freeze_schedule.py の frozen_at_epoch 契約より*

- 内部状態を `_is_frozen: bool` から `_frozen_layers: set[int]`（累積）+ `_last_frozen_layer` へ昇格。`is_frozen` / `frozen_layer_idx` はこの状態から再計算し、単層テストの契約を維持。
- `_freeze_layer` に凍結の共通処理を集約。`apply_freeze`（単層・解決済み対象）と `apply_freeze_layer`（progressive・明示層）の両方が使用。
- `_capture_xin_for_layer` / `_capture_live_layer_input` に xin 取得（no-grad）と活性取得（grad 保持）をレイヤ指定でファクタリング。単層 `cache_xin` / `compute_local_loss` はこれらを呼び出す薄いラッパとして挙動を完全保持。

### 層別 xin キャッシュと layer_idx 選択 🔵

**信頼性**: 🔵 *progressive_freeze.py compute_local_loss・_capture_xin_for_layer 実装より*

- `_xin_caches: dict[int, dict[int, list[Tensor]]]` で層別に xin を保持。`compute_local_loss(layer_idx=L)` は `_xin_caches[L]` と層 L の現在入力をペアにする（手計算 MSE で xin(4) と誤って xin(5) を選ばないことを検証）。
