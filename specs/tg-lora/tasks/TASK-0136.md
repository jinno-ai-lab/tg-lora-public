## タスク概要

steering は前反復の M10.3 disk-death guard を「VALUABLE」と評価しつつ、ひとつの自己導入回帰を指摘した。feature コミット群（`3c2c9d7`/`db3c08f`）が `keep_last_checkpoints` / `min_free_disk_gb` を M10 YAML に書き込んだ時、これらの knob はまだ `LoggingConfig` に宣言されておらず、同 model は `extra="forbid"` であるため **M10 config は発射不能**になった。これは follow-up fix コミット `d2218ed` でのみ修正された — つまり feature コミットが **shipped config を壊したまま main に乗った**。steering の教訓は明示的: 「launch check（schema round-trip / Hydra resolve）を新 LoggingConfig knob 追加の直後・同一コミットで検証せよ」。

本タスクはこの教訓を「M10 を覚えておく」ではなく**機構**へ昇格する。現状、発射性 round-trip で守られているのは schema-valid 32 config 中わずか 6（accel 4 + M10 2）にすぎない。残り 26（mainline `9b_tg_lora.yaml`・`9b_baseline.yaml`・paper-PoC 群・prefix-cache 群・jsonex 群・measure 群 を含む）には発射性テストがなく、将来の任意の `LoggingConfig` 変更（knob 追加・rename・制約強化・削除）がこれらを壊しても feature コミットでは何も検知しない — d2218ed クラスの一般化された再発経路である。

解決策は単一の自動関門: 出荷された **全** training config が `load_and_validate_config` を通って `BaselineConfig | TGLoRAConfig` へ round-trip することを要求する。これにより schema 変更が config を壊した瞬間、それを導入したコミットと同一コミットで失敗する。5 つの `mlx_*` config は Apple-Silicon MLX（`mlx-lm`）向けであり Pydantic 経路ではなく `mlx/scripts/*` の別 launch path を使うため、原則的に除外し、除外集合はそれ自身の「正直さ」テストで固定する（陳腐化・typo も即失敗）。

## 主要実装項目

### repo-wide 発射性 parametrize（`test_config_launchability_gate.py`）🔵

**信頼性**: 🔵 *`load_and_validate_config`（config_schema.py L506）が OmegaConf resolve→Pydantic 検証の実ゲートであり、accel 先例テストと同一 idiom より*

- `CONFIGS_DIR.glob("*.yaml")` で出荷 config 全件を収集し、`MLX_LAUNCH_PATH_CONFIGS`（5 件）を除いた全件を `@pytest.mark.parametrize` で round-trip。各 config が `BaselineConfig | TGLoRAConfig` へ解決することを検証。
- 32 config 中 26 が本タスクで**初めて**発射性テストの傘下に入る（mainline `9b_tg_lora.yaml` / `9b_baseline.yaml` / paper-PoC 群 / prefix-cache 群 / jsonex 群 / measure 群）。これまでこれらは schema 変更に対して無防備だった。

### 除外集合の正直さ固定（同ファイル）🔵

**信頼性**: 🔵 *各除外 config が実際に `ValidationError` を投げることは本 worktree で実測済み（26/38/45/27/27 errors for BaselineConfig）/ `mlx-lm` が `mlx/scripts/*` 別 launch path であることは Makefile + mlx/ tree より*

- `MLX_LAUNCH_PATH_CONFIGS` は「脱出ハッチ」だが、それが陳腐化・typo しないよう二つのテストで固定: (1) 全除外名が disk に存在、(2) 全除外 config が本当に検証失敗する。後者は**後に修正された除外 config**（今は validate するようになった）を即座に検知し、隠蔽されず除外リストから削除させる。
- これにより「新 config が黙って失敗する」余地をゼロにする: 新 config は要么 緑の parametrize 傘下に入り、要么 意識的に除外リストへ理由付きで追加される。

### d2218ed クラスを捕える不変量の pin（同ファイル）🔵

**信頼性**: 🔵 *`LoggingConfig` の `model_config = ConfigDict(extra="forbid")`（L199）を直接検証 / d2218ed がまさにこの `extra="forbid"` で発火した実史より*

- `test_logging_config_rejects_undeclared_keys` が `LoggingConfig(run_dir=..., bogus_undeclared_knob=1)` で `ValidationError` を要求。これは「なぜ関門が d2218ed クラスを捕えるか」を文書化するだけでなく、将来 `extra="ignore"` へ緩和する変更（関門を無力化し回帰クラスを再開する）を即座に失敗させる番兵でもある。

## AI Hub steering との整合

- **bullet（launch check を feature コミットと同一コミットで）**: これを「M10 を再テストする」でなく機構へ昇格した。出荷 config 全件を傘下に入れる単一自動関門により、次に誰かが `LoggingConfig`（または任意の config model）へ knob/rename/制約を加えた時、壊れた config を導入コミットと同一コミットで検知する。d2218ed は「feature が壊し follow-up fix が直した」だったが、以後その follow-up は不要になる。
- **judge_unavailable（agent-proxy LLM backend なし）**: 判断経路が LLM に依存しない事実ベースの検証（Pydantic round-trip・mutation 再現・`extra="forbid"` 実証）で成立。LLM 判断不在でも検証可能性は完全。
- **North Star（docs/specs だけ増やして進捗扱いしない）**: TASK-0136 は要件定義（本ファイル）と**同時に**その完了判定を実装（`test_config_launchability_gate.py`）・検証（35 緑 + mutation）した。spec の完了条件はすべて `[x]` であり、検証コマンド（`pytest tests/test_config_launchability_gate.py`）と直結する。
