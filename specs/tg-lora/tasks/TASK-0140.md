## タスク概要

Guard 実験（10_guard_experiment）の事後解析器 `scripts/analyze_dynfreeze_experiment.py` は、§5.2 二段階停止プロトコルを post-hoc で解決する: L*（baseline の最良品質損失）と G*（その cycle の gold_combined）を baseline から読み、Guard 条件 B が「品質損失 ≤ L*+margin AND gold ≥ G*」に到達した最初の cycle を停止点とする。この停止判定が実験全体の quality/speed verdict を決める核心である。

ところが、**L* の抽出元（`extract_loss_and_time`・`baseline_targets`）・一次トリガの損失比較（`guard_stop_point`）・gold 行の損失列（`extract_gold`）は、すべて pilot/split 境界 proxy 損失 `loss_valid` を読んでいた**。設計 §5.1/§5.2/§5.3 とモジュール docstring は揃って **full-eval 損失 `valid_full`** を品質信号に指定する。`loss_valid` は毎 cycle の安価な proxy であり、停止判定の根拠としては §7 鉄則の「中間指標で結論しない」に抵触する。

本タスクは、この誤参照を **pluggable-receiver パターン（§6.2/§6.3 が scale 測定で用いる「受口を建てておき、誠実な計測が届けば活性化・届くまでは byte-identical」）** で閉じる:

1. **full-eval 損失定数と受口**: `LOSS_VALID_FULL_KEY = "loss_valid_full"` を追加。`extract_loss_and_time(*, full_eval_only=False)` は flag 時 `loss_valid_full` のみを、default は `loss_valid` を読む。`extract_gold` の行損失は `loss_valid_full` があればそちらを、無ければ `loss_valid` に fallback。
2. **L* の誠実化**: `main()` は baseline の full-eval 曲線を追加抽出し、`write_gate_decision` に `baseline_full_losses/baseline_full_cycles`（末尾 default `None`）として渡す。`write_gate_decision` 内 L* 計算は full-eval 曲線を優先し、空なら proxy 曲線に fallback（byte-identical）。
3. **零カバレッジの閉包**: 決定関数 4件（`baseline_targets`・`guard_stop_point`・`extract_gold`・`extract_loss_and_time`）は本タスク前 直接テスト 0件だった。新設 `TestQualityGateDecision`（17件）が契約を pin し、verdict-flip 端到端テストで「proxy では停止・full-eval では非停止」という修正の効果を証明する。

trainer（`train_tg_lora.py`）側は GPU 専用で本環境で非実行可能であり、`loss_valid_full` を step record に永続化していない（full-eval 損失は計算して `cycle_state.record_full_eval` に渡るのみ）。ゆえに本タスクは **受口（analyzer）のみを是正** し、trainer からの誠実データ供給は別課題（Category C・GPU 依存）とする。これは §6.x の pluggable-receiver 誠実性パターンそのものであり、「受口は正しい・データが届けば即活性化・届くまでは legacy と byte-identical」を保証する。
