## タスク概要

§5.2 quality/speed gate は **A/B 比較**である: L\*（最良品質損失）は baseline (A) run から、停止点の品質損失は guard (B) run から読む — ともに `loss_valid_full`（full-eval）優先・`loss_valid`（pilot proxy）fallback の同一フィールドから。TASK-0140 はこの両経路を full-eval 優先化し、TASK-0143 は dormant run（`loss_valid_full` を 1 件も運ばない run = 現状の全実 run）が silent byte-identical fallback で L\* を proxy 上に置く状態を、`honesty_contract_status` + DORMANT 警告で可視化した。

しかし **TASK-0143 の可視化は baseline 側のみ**だった: `main()` は `honesty_contract_status(baseline_records)` のみ計算し `write_gate_decision(honesty_status=baseline_honesty)` のみ渡した。guard records は一度も診断されない。これは決定的な帰結を持つ: baseline run が GPU で再生成されて ACTIVE になっても、guard run がまだ旧ファイルなら guard 側の停止点損失は proxy にフォールバックし、fair speed 比較は片側汚染されたまま — レポートは「ACTIVE」と表示し警告を出さない。これは TASK-0143 が baseline 側で排除した静かな不活性の、比較のもう半分である。

本タスクはこれを閉じる: 比較は **両側が ACTIVE でのみ ACTIVE**（片側でも dormant なら汚染）とし、`dormant_sides`（`"A"` baseline / `"B"` guard）で汚染元を名指しする。診断は fail-conditioned で、TASK-0141 が trainer を閉じて `loss_valid_full` が永続化されれば全 run で ACTIVE に反転し警告は自己消去する（TASK-0142/0143 と同一の自己陳腐化防止設計）。実装は 3 層:

1. **結合層** (`comparison_honesty_status(A,B)`): 両 per-run 状態を取り、dormant な側のリストと比較 state（両 ACTIVE のみ `active`）を返す。両 `None` なら `None`（診断未導入前と byte-identical）。片側のみ与えられればその側が単独で verdict を駆動（既存の baseline-only 呼出・テストと後方互換）。
2. **可視化層** (`format_comparison_honesty_line`): dormant → `⚠ Honesty contract DORMANT on <A|B|A and B>: … pilot proxy … A/B time-to-quality 比較は汚染 … TASK-0141` / active → `ACTIVE on both A and B (A M/N; B M/N) … full-eval loss`。決して空行を返さない。
3. **配線層** (`main()` + `write_gate_decision`): `main()` が両 run を診断し比較行を stdout に印字。`write_gate_decision` の `guard_honesty_status=` は optional（default `None`）で、`comparison_honesty_status(honesty_status, guard_honesty_status)` が `None` でなければ比較行を `gate_decision.txt` に永続化。
