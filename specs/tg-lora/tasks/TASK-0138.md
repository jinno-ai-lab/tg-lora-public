## タスク概要

M10.3 disk-death guard ファミリ（db3c08f → 3c2c9d7 → d2218ed → 43acef0 → c96a5a0 → 693ac84）は、TG-LoRA 学習経路（`train_tg_lora`）上の**2つの無限蓄積ベクトル**を閉じた（cycle-checkpoint ディレクトリ・trajectory-delta-artifact ファイル）。しかし本 guard ファミリは、**もう1つの学習エントリポイントであるベースライン QLoRA 経路（`train_baseline_qlora`・`make train-baseline`）を一切カバーしていなかった**。

`train_baseline_qlora` は周期保存で `checkpoint-<global_step>` ディレクトリを `save_every_steps`（既定 250）毎に書き（L493-496）、`save_trajectory_delta_artifacts: true`（例: `9b_baseline_suffix_only_last25.yaml` L37・`9b_baseline_with_prefix_cache.yaml` L36 が出荷）のとき `save_trajectory_delta_artifact` でステップ毎に `.pt` を `trajectory_delta_artifacts/` へ書く（L414）。これらを**削除する経路はベースライン経路上どこにもない**。

既存 cycle 保護 `prune_checkpoint_cycles` の regex は `^checkpoint-cycle-(\d+)$`（`_CYCLE_DIR_RE`）であり、`checkpoint-<step>`（`-cycle-` infix なし）には**意図的にマッチしない**。よってベースライン経路のステップ checkpoint は cycle 保護の傘外であり、trajectory-artifact 保護（TASK-0137）も `train_baseline_qlora` からは**呼ばれていなかった**。長時間のベースラインランやスイープは、保護が「存在するのにこのコードパスでは発火しない」クラスのまま M10.3 インシデントクラスを再生する。

本タスクはこの残渣を閉じる。新規 knob は増やさない — TG-LoRA 経路と**同一の `keep_last_checkpoints` / `min_free_disk_gb`** を読む `prune_step_checkpoints(_from_cfg)` を新設し、既存の `prune_trajectory_delta_artifacts_from_cfg` と共にベースライン周期保存経路へ接続する。`checkpoint-<step>` を識別する regex `_STEP_DIR_RE = ^checkpoint-(\d+)$` は `_CYCLE_DIR_RE` と**構造的に素**（`checkpoint-cycle-5` は `checkpoint-` の直後が数字でないためマッチせず、逆も然り）であり、両方向の非干渉テストで固定する。3つのベクトル全てが共通の `_bound_keyed_paths` core（TASK-0137 抽出）で同一方針を共有する。

## 主要実装項目

### `prune_step_checkpoints` + `_sorted_step_checkpoint_dirs`（`checkpoint.py`）🔵

**信頼性**: 🔵 *TG-LoRA 経路の `prune_checkpoint_cycles` / `_sorted_cycle_checkpoint_dirs` と機械的に対称 / 共通 `_bound_keyed_paths` core を使用し同一方針を共有 / 既存テストパターンを直接踏襲*

- `_STEP_DIR_RE = ^checkpoint-(\d+)$` でベースライン経路の `checkpoint-<global_step>` を識別。`prune_checkpoint_cycles` と同一契約（`keep_last` で最新 N 件の step dir を残し、`min_free_disk_gb` で oldest-first に回収・単一最新 step は絶対残す）。dirs は `shutil.rmtree(ignore_errors=True)`。`_bound_keyed_paths` を経由するため、3ベクトル全てが同一の count-bound + disk-floor 方針を共有し、方針の分岐バグを防ぐ。

### `prune_step_checkpoints_from_cfg` seam（同ファイル）🔵

**信頼性**: 🔵 *`prune_checkpoint_cycles_from_cfg` と同一の cfg 読取・knob 正規化・両 OFF で noop の契約 / plain dict・OmegaConf 両対応*

- ベースライン周期保存経路への接続点。TG-LoRA 経路を抑えているのと**同一 knob** を読むため、1つの opt-in で両エントリポイントの全ベクトルを抑える。出荷ベースライン config は default-off のため、opt-in までは今日の無限蓄積挙動を保存（比較/paper ランの挙動を変えない）。

### ベースライン周期保存経路への両 seam 接続（`train_baseline_qlora.py`）🔵

**信頼性**: 🔵 *`save_baseline_training_state` 直後の周期保存 block に step 保護・artifact 保護を追加 / `train_tg_lora.py` の周期保存 block（TASK-0131/0137 接続）と対称な構造*

- `for d in prune_step_checkpoints_from_cfg(cfg, run_dir)`・`for f in prune_trajectory_delta_artifacts_from_cfg(cfg, run_dir)` を順に呼び、pruned を `logger.info` で記録。両 knob が 0 なら両者とも noop（今日の挙動を保存）。

### 両方向非干渉の固定（テスト）🔵

**信頼性**: 🔵 *regex の素性を実行時テストで固定 / `test_step_prune_never_touches_cycle_dirs`（step 保護が cycle dir を触らない）・`test_cycle_prune_never_touches_step_dirs`（cycle 保護が step dir を触らない）*

- run_dir に `checkpoint-{step}` と `checkpoint-cycle-{N}` を混在させ、各保護が相手のリカバリポイントを絶対に削除しないことを検証。`checkpoint-<digits>` と `checkpoint-cycle-<digits>` の命名空間は構造的に素。
