## タスク概要

AI Hub feedback（前々イテレーション）は「gate を failing 状態で放置せず、claim を進めるか gap を正式記録せよ」と指示した。TASK-0142 は**反証 FAIL** の自己記述（known-limitation）で後者を閉じた。しかし feedback が一貫して警告してきた「静かな状態」の残り半分は、**未測定 gate が反証として読める**ことだった: external-eval を渡さず `evaluate_paper_gates.py` を走らせると G3 は `passed: False` → report "AT LEAST ONE GATE FAILED" → exit 1 となり、まだ一度も G3 実験をしていない run が「G3 の品質保持 claim は反証された」と主張しているかのように見える。

本タスクはこれを**検証可能な振る舞い差分**として閉じる（doc-only ではない）。gate が必須入力の欠落で bail したとき、`passed: False` は維持しつつ（下流 `consolidate` 互換）新たに `evaluated: False` を立て、known-limitation を反証レコードではなく `status="insufficient_evidence"` レコードに切り替え、report に `INSUFFICIENT EVIDENCE`（`ℹ`）第3状態を表示し、default exit code を反証 FAIL 限定にする（`--strict` で従来挙動を復元）。これは「未測定 ≠ 反証」という科学的誠実性そのものであり、§5.2 の「proxy fallback ≠ honest data」と同型の契約を第2評価器に拡張した。

副次的に、generic fallback owner の誤帰属を修正した: TASK-0142 は `_KNOWN_LIMITATION_OWNER`（TASK-0142 = G1/G4 専用）を全 gate の generic fallback に使っていたため、G0/G2/G3 の disproven gap が G1/G4 task に誤帰属する潜在バグがあった。G1/G4 の custom builder は引き続き TASK-0142 を返す（既存テストが pin 済み）が、generic path は新設 `_KNOWN_LIMITATION_OWNER_GENERIC`（`master_plan §2.2`）を使う。

実装は 5 層:
1. **状態層** (`evaluated` flag): `_check_g3`/`_check_g4` の入力欠落 early-return に `evaluated: False`。他は default `True`（`.get("evaluated", True)`）。
2. **導出層** (`_known_limitation_for`): `evaluated=False` を最初に検出し `insufficient_evidence` レコードを返す（反証レコードの前）。`missing_input` は `_GATE_REQUIRED_INPUT` から gate 名で引く。
3. **可視化層** (`_format_report`): `evaluated=False` → `INSUFFICIENT EVIDENCE` + `ℹ Insufficient evidence` ブロック。FAIL は `⚠ Known limitation` のまま。
4. **終了層** (`main()`): default は disproven FAIL（`evaluated and not passed`）のみ exit 1。`--strict` は insufficient も exit 1（従来の fail-unless-arrived）。insufficient gate は Note 行で名指し。
5. **機械可読層** (JSON): `overall_passed`（evaluated gate のみ）・`disproven_fail_gates`・`insufficient_evidence_gates` を分離。
