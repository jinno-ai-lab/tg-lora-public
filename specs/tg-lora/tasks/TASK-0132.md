## タスク概要

`LEVEL1_REALIZED_REDUCTION_CEILING = 0.0` は CPU-proxy in-vivo 証拠に基づく境界であり、design §6.2 と ceiling コメント（L279-281）は共に「将来の in-vivo 計測で Level-1 に非零の実現削減が観測されれば（例: gradient checkpointing 併用下）、この ceiling を引き上げて回復させる」と明記していた。しかし実装側にはその「引き上げ」を受け取る経路が存在せず、定数の手書き修正以外に方法がなかった。AI Hub steering（bullet 1）も同じ指摘: 「非零の Level-1 in-vivo 削減が観測されれば `LEVEL1_REALIZED_REDUCTION_CEILING` を 0.0 超に引き上げて回復せよ」。

本タスクは、その回復を**監査可能な証拠経路**として実装する。`Level1RealizationRecord` が1件の計測証拠（観測値・ラン数・source）を運び、`resolve_level1_ceiling()` が関門の信用する ceiling 値を解決し、それが verdict スタック（`realizable_reduction` → `gate_reduction` → `speed_gate_verdict` → `compare_freeze_levels`）を通って Level-1 判定を `FAIL` から回復させる。既定（record なし）は検証済み `0.0` を維持し、**一切の既存判定を変化させない**。これは gate の厳格化ではなく、実測値が得られた時にそれを反映する pluggable な着地点である。

## 主要実装項目

### 証拠レイヤ（Level1RealizationRecord / resolve_level1_ceiling）🔵

**信頼性**: 🔵 *§6.2 design「ceiling を引き上げて回復させる」+ ceiling コメント L279-281 + §6.3 と同値の薄証拠 bar より*

- `Level1RealizationRecord` は `ReductionSample` の §6.3 honesty 哲学を ceiling に反映: 観測値・ラン数・`source`（例 `9b_cuda_grad_ckpt`）を運び、回復の根拠を監査可能にする（手書き定数修正でない）。
- `resolve_level1_ceiling` は「薄い証拠（3件未満）は検証済み `0.0` を維持」を強制。1-2件の非零観測で Level-1 FAIL が黙って PASS に反転しない。

### verdict スタックへの伝播（level1_ceiling / level1_record）🔵

**信頼性**: 🔵 *realizable_reduction / gate_reduction / speed_gate_verdict / compare_freeze_levels の既存契約より*

- 算術レイヤ（`realizable_reduction`/`gate_reduction`/`speed_gate_verdict`）は解決済み float `level1_ceiling` を受け取る。実験レイヤ（`compare_freeze_levels`）は record オブジェクト `level1_record` を受け取り1度解決する。
- `realized = min(ceiling, proxy)` により、観測値が算術上限を超えても算術値で打ち切る（実現削減率が算術削減率を超えることは物理的にない）。
- `LevelComparison.level1_ceiling`（既定 `0.0`）が「検証済み基線か、計測で回復された値か」を監査に残す。
