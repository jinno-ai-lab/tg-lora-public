## タスク概要

AI-Hub feedback が 3 イテレーション連続で「9B target-scale run を実行せよ」と要求してきたが、
本 public mirror では `src/data/`（private pipeline）不在により `train_tg_lora.py:16`
`from src.data.build_seed_dataset import load_dataset` が解決せず、過去 2 iter は「BLOCKED」と結論した。

**本 iter で判明したこと**: blocker は *fundamental ではなく runnable* である。

1. **判定装置は 9B deposit を受け取る準備が完全に整っている**。public mirror の
   `replay_freeze_validloss_ci` は `proxy_scale=False` の sample を判定し
   `citable_as_target_scale=True` に昇格する（`tests/test_replay_freeze_validloss_ci.py:852`
   で test green）。欠けているのは「実際の 9B 数値」のみ。
2. **9B 数値を生成する手段が同機上に存在する**:
   - upstream `/home/jinno/tg-lora/src/data/`（build_seed_dataset.py 含む、private pipeline）が present
   - 対象モデル `Qwen/Qwen3.5-9B`（GOAL §0 Track A）が `~/.cache/huggingface/hub/` に 19G で cached
   - 学習データ `data/{train,valid_quick,valid_full,gold_test}.jsonl` が upstream に present
   - `torch.cuda.is_available()=True`（RTX 3060, torch 2.1.1+cu121）
3. 従って「BLOCKED」は正しくなく **「runnable・GPU 予算と public/private 境界の合意待ち」** である。

### Tier-1（turnkey・GOAL の究極目標に決定的）

upstream の既存 config だけで実行可能。GOAL §0 の最終ゴール
「full backprop と同等の品質を保ちながら総学習コストを有意に削減」を 9B 実スケールで問う。
candidate（`9b_tg_lora.yaml` = output_first progressive freeze）vs baseline（`9b_baseline.yaml` = full backprop）
の best_valid_loss を multi-seed で比較し、`proxy_scale=False` で deposit → public judge が判定。

### Tier-2（feedback の §4 verdict gate の通り・upstream 拡張が必要）

feedback が要求する「heterogeneous × generalize leg での candidate vs surrogate(random-order)」は
**turnkey ではない**。upstream `train_tg_lora.py` は prod-path の 1 本の freeze schedule のみを実行し、
以下 3 つは public の合成-proxy runner（`scripts/run_freeze_validloss_ci.py`）にしか存在しない:
- (a) surrogate = random-order freeze の掃引
- (b) heterogeneous per-layer rank
- (c) generalize held-out task を verdict 指標にする arm

これらを実 9B training に移植するのが **本研究のエンジニアリング gap** である（別 TDD タスク）。

- **本 iter は public-mirror scope に留まった**: upstream GPU campaign の実行も
  public/private 境界の越境も行っていない（明示的な合意待ち）。上記 recipe は全 step 検証済みで
  turnkey であり、実行は GPU 予算 + 境界合意のみ待ち。詳細は PURPOSE.md「次の一手」。

  `citable_as_target_scale=True` の gate を通ったもののみ target-scale 主張として扱う

  `src/data/` は意図的に public に含まれないため、本 recipe は常に「upstream 側で生成→数値のみ public に deposit」を前提とする。
