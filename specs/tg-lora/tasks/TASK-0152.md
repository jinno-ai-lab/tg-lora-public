## タスク概要

AI-Hub feedback が 3 イテレーション連続で「9B target-scale run を実行せよ」と要求してきたが、
本 public mirror では `src/data/`（private pipeline）不在により `train_tg_lora.py:16`
`from src.data.build_seed_dataset import load_dataset` が解決せず、過去 2 iter は「BLOCKED」と結論した。

**本 iter で判明したこと**: blocker は *fundamental ではなく runnable* である。

1. **判定装置は 9B deposit を受け取る準備が完全に整っている**。public mirror の
   `replay_freeze_validloss_ci` は `proxy_scale=False` の sample を判定し
   `citable_as_target_scale=True` に昇格する（`tests/test_replay_freeze_validloss_ci.py:852`
   で test green）。欠けているのは「実際の 9B 数値」のみ。
2. **9B 数値を生成する手段が同機上に存在する**:
   - upstream `/home/jinno/tg-lora/src/data/`（build_seed_dataset.py 含む、private pipeline）が present
   - 対象モデル `Qwen/Qwen3.5-9B`（GOAL §0 Track A）が `~/.cache/huggingface/hub/` に 19G で cached
   - 学習データ `data/{train,valid_quick,valid_full,gold_test}.jsonl` が upstream に present
   - `torch.cuda.is_available()=True`（RTX 3060, torch 2.1.1+cu121）
3. 従って「BLOCKED」は正しくなく **「runnable・GPU 予算と public/private 境界の合意待ち」** である。

### Tier-1（turnkey・GOAL の究極目標に決定的）

upstream の既存 config だけで実行可能。GOAL §0 の最終ゴール
「full backprop と同等の品質を保ちながら総学習コストを有意に削減」を 9B 実スケールで問う。

`proxy_scale=False` で deposit → public judge が判定。

> **⚠️ 2026-06-29 arm 訂正**: 旧版の「candidate = output_first progressive freeze」は誤り。
> `9b_tg_lora.yaml` の active block は `tg_lora:`（K/N/alpha/beta velocity・layer sampling・
> rollback/accept gate）のみで、progressive freeze schedule は不在。よって本 Tier-1 は
> **TG-LoRA vs full-backprop baseline**（= §0/§3.1 の「品質保持×コスト削減」を 9B で問う）
> であり、§4 の candidate-vs-surrogate(random-order freeze) verdict は **Tier-2（別 TDD）**。

### Tier-2（feedback の §4 verdict gate の通り・upstream 拡張が必要）

feedback が要求する「heterogeneous × generalize leg での candidate vs surrogate(random-order)」は
**turnkey ではない**。upstream `train_tg_lora.py` は prod-path の 1 本の freeze schedule のみを実行し、
以下 3 つは public の合成-proxy runner（`scripts/run_freeze_validloss_ci.py`）にしか存在しない:
- (a) surrogate = random-order freeze の掃引
- (b) heterogeneous per-layer rank
- (c) generalize held-out task を verdict 指標にする arm

これらを実 9B training に移植するのが **本研究のエンジニアリング gap** である（別 TDD タスク）。

## 実装詳細

### Tier-1 — turnkey 計測（全コマンド本 iter で検証済み）🔵

**信頼性**: 🔵 *upstream Makefile / config / data / model cache / public replay CLI を全て直接確認*

> **⚠️ 2026-06-28 実測補正（"seq_len=1024 turnkey" は誤り）**: 上記 step1/2 の素の
> `make train-*` は **12GB RTX 3060 で OOM する**。pin 済み `data.max_seq_len=1024`
> （および 512）で CE-loss 経路 `ForCausalLMLoss→logits.float()` が ~3.8GiB / ~1.9GiB
> を要求し、空き ~2.2GiB に収まらない（9B 4-bit 重みが ~9.4GiB を占有）。**動作する
> VRAM-floor config（実測・OOM なし・loss 1.15→1.03 で学習）**: `data.max_seq_len=256`
> + `eval.eval_batch_size=1`（既定 16 が eval でバッチ化して OOM）+
> `PYTORCH_CUDA_ALLOC_CONF=expandable_segments:True`。seed override は
> `experiment.seed`（`training.seed` は pydantic extra_forbidden）。`logging.run_dir`
> は timestamp ではなく config 指定の固定パス → seed 毎に上書き必須（clobber 防止）。
> よって本 Tier-1 計測は **reduced-context(seq256)・reduced-budget の probe** であり、
> GOAL §4 の完全 seq_len=1024 verdict（>12GB VRAM 必要）ではない。tag は deposit に
> 正しく付与する。詳細と campaign driver は PURPOSE.md「次の一手 / audit」。

#### 1. multi-seed candidate（plain TG-LoRA velocity extrapolation）🔵

```bash
# upstream repo（private pipeline）で実行。seed を掃引。
# ⚠️ 素の make train-tg-lora は seq_len=1024 で 12GB OOM する（上記 ⚠️ 実測補正）。
#    動作する VRAM-floor 起動 = module 直呼び + seq256 + eval_batch_size=1 + expandable_segments。
cd /home/jinno/tg-lora
export PYTORCH_CUDA_ALLOC_CONF=expandable_segments:True
PY=/home/jinno/tg-lora/.venv/bin/python
for SEED in 42 43 44; do
  "$PY" -m src.training.train_tg_lora --config configs/9b_tg_lora.yaml \
    --override experiment.seed="${SEED}" \
    --override training.max_cycles=15 \
    --override data.max_seq_len=256 \
    --override eval.eval_batch_size=1 \
    --override logging.run_dir="runs/9b_verdict/cand_seed${SEED}"
  # -> runs/9b_verdict/cand_seed${SEED}/run_metrics.jsonl の run_footer.best_valid_loss
done
```

**実装ファイル（参照元）**: upstream `src/training/train_tg_lora.py`（`load_dataset`@L763 /
valid_loss emit@L615 `run_metrics.jsonl`）、config `configs/9b_tg_lora.yaml`（`name_or_path: Qwen/Qwen3.5-9B`, `seed: 42`）。
**compute match**: candidate の `max_cycles=15` は footer 実測で optimizer_steps が
seed で変動する（cand_seed42=94 / cand_seed43=87 / cand_seed44=61）— 外挿受容率依存。
`full_backward_passes` も連動して変動（752 / 696 / 488・mean 645）。baseline は
`max_steps=94`（grad_accum=8 → 752 backward）で固定。よって compute-match は optimizer-step
上限（94）で合わせ、candidate の bw は自然値（外挿成功時に baseline を下回る = コスト削減の現れ）。

#### 2. multi-seed baseline（full backprop・compute-matched）🔵

```bash
cd /home/jinno/tg-lora
export PYTORCH_CUDA_ALLOC_CONF=expandable_segments:True
PY=/home/jinno/tg-lora/.venv/bin/python
for SEED in 42 43 44; do
  "$PY" -m src.training.train_baseline_qlora --config configs/9b_baseline.yaml \
    --override experiment.seed="${SEED}" \
    --override training.max_steps=94 \
    --override data.max_seq_len=256 \
    --override eval.eval_batch_size=1 \
    --override eval.full_eval_every_steps=10 \
    --override eval.quick_eval_examples=32 \
    --override logging.run_dir="runs/9b_verdict/base_seed${SEED}"
done
```

> **eval-cadence fix（2026-06-29 実測・必須）**: baseline trainer は
> `global_step % cfg.eval.full_eval_every_steps`（既定 250）の時しか eval せず、`max_steps=94 < 250`
> では**eval が一度も発火せず** `loss_valid` が記録されない → form CLI が ValueError。よって
> `--override eval.full_eval_every_steps=10`（step 10,20,…,90 で 9 点評価）が**必須**。
> 併せて `--override eval.quick_eval_examples=32` で candidate（cycle 毎 32-example quick_eval）と
> **eval sample size を一致**させ apples-to-appys とする（baseline 既定 64 ではなく 32）。
>
> **eval-regime caveat（訂正版 2026-06-29）**: 旧版の「baseline は ~454-example full-eval」は**誤り**。
> baseline の周期 eval も `quick_eval_examples`（32 に統一）に cap される — candidate と**同一規模の quick_eval**
> であり、454-example full-eval ではない（`train_baseline_qlora.py:458` の `max_examples=cfg.eval.quick_eval_examples`
> で実証）。両 arm とも 32-example quick_eval で variance は同等。完全 §4 verdict には
> seq1024 + `valid_full`(493) による統一 full-eval が必要（GOAL §3.3 L191 契約）。
