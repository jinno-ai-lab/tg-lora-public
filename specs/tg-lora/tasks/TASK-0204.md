## bug（end-to-end 再現可能）

`CycleMonitor.detect_divergence` の spike 検出は `train_loss / prev_train >= spike_threshold` のみを検査していた。`valid_loss` は `current.get("valid_loss")` で取得済み（NaN/Inf 検査と `update` の best-tracking で使用）だが、spike 検査には使われなかった。

これは2重の不整合:
1. **docstring promise 違反** — モジュール docstring は「Detects divergence (loss spikes, NaN gradients)」と**汎用の "loss spikes"** を謳うが、実装は train_loss のみ（TASK-0200 と同じ docstring-promise 署名）。
2. **monitor 内部の loss 権威性との不整合** — `update` は best-tracking で `loss = valid_loss if valid_loss is not None else train_loss`（valid_loss を権威的 loss として選択）するが、`detect_divergence` は spike 検出だけ train_loss に固定。同じ「run は発散しているか？」の問いに2つの異なる信号を使う。

**結果**: train_loss が安定（1.0→1.0）のまま valid_loss が 3x 急増（0.5→1.5）する汎化崩壊サイクルは `detect_divergence` を素通りし、`CycleMonitor.update` は `status="healthy"` + `recommendations=[]`（advisor 経由では `no_action`）を返す。`advise_training` CLI（producer の `run_metrics.jsonl` を読み、`_extract_cycle_records` が `valid_loss` を monitor へ stream）では、この発散 run が healthy として報告される — feedback が TASK-0200 で問題視したのと同形（発散 run の healthy 報告）。

`valid_loss` は producer が per-cycle で記録する（`record_step(loss_valid=loss_pilot, ...)`、full-eval cycle では `loss_valid_full` が merge）ため、実 producer 出力から到達可能。

## 修正

spike ループを `train_loss` のみから `(train_loss, valid_loss)` の双方に拡張:

```python
prev_valid = previous.get("valid_loss")
for name, cur, prev in (
    ("train_loss", train_loss, prev_train),
    ("valid_loss", valid_loss, prev_valid),
):
    if cur is not None and prev is not None and prev > 1e-6:
        if cur / prev >= self.spike_threshold:
            return DivergenceReport(detected=True, metric=name, severity="high",
                                    current_value=cur, threshold=prev * self.spike_threshold)
```

- `train_loss` を先に検査 → 両方 spike 時に train が `metric` label を勝ち取る（上記 NaN/Inf ループの順序と同一）。
- 各信号に `prev > 1e-6` near-zero guard（既存 train guard と同一・ratio 増幅の false-positive を防止）。
- `valid_loss` が `None`（loss-only history・古い fixture）のときは valid entry を skip → train-only 挙動と bit-identical（`test_valid_absent_falls_back_to_train_only` で pin）。
- severity は `train_loss` spike と同一の `"high"`（保守的・過発火回避）。NaN/Inf のみ `"critical"` という既存語彙を維持。

## 意義と非自明性

- TASK-0200（grad_norm）と**同じ divergence detector**上の別軸だが、性質が異なる: grad_norm は「dead read（取得して捨て）」、本 TASK は「half-wired（valid_loss は取得して NaN 検査に使うが spike 検査には使わない）」。feedback が明示的に「dead-read/dropped **または** half-wired 未実装」を hunt 対象に含めた両クラスをカバー。
- **trajectory の z-score 異常検査（`detect_anomalies`）との非冗長性**: trajectory は `min_points`(>=3) かつ z-score>3.0（統計的外れ値）を要し、`_losses()`（valid 優先）上で動く。本 monitor spike 検査は**2 cycle の ratio>=2.0**（連続サイクル間の急増）で、点数要件なくより早期に発火 — 相補的。
- **過早枯渇宣言の是正**: TASK-0199 の広域 sweep「非禁制 lane に bug 不存在」を5度目に覆す。silent-success surface は未枯渇。
