## 背景（launch-ready leg の発火が露呈した真の robustness bug）

TASK-0161 で open leg（heterogeneous × generalize × **full-budget**・1500 step）を launch-ready に格上げしたが、配線のみで auto-fire しなかった（GPU contended + hard-to-reverse outward action）。本 iter は feedback lever #1（9B run unblock = 最優先）に従い stable dir から**実際に発火**した。その際、contended GPU 下で**真の robustness bug が露呈**した:

- pre-flight `gpu_free_memory_deferred`（`--min-free-gib` snapshot gate）は attempt 時の空きで **PASS** するが、~14 秒後の `load_base_model` → `Qwen3_5TextModel.__init__` → `torch.empty(device=cuda)` までの間に**並行 private run が GPU を再占有** → uncaught `torch.AcceleratorError: CUDA error: out-of-memory` → **exit 1 (FATAL)** → launcher が loop 終了（max_attempts/deadline を待たず task 自体が死）。
- これは TOCTOU race（snapshot→allocation 間の reclaim）であり、pre-flight gate では捕捉不能。既存 `OutputPathDiedDuringRun` handler と同じ「run-time に外部状態が変わる」class だが、OOM は**再試行可能**（窗口が開けば成功する）なのに **fatal として分類**されていた = launcher が生き残れない唯一の bug。
- 従来の homogeneous full run（`4b88ca8`）は GPU が充分空いていたためこの path に到達せず、bug は潜在化していた。heterogeneous run は private run と窗口を争うため初めて発火。

## 実装（run-time CUDA-OOM を tempfail exit 75 に分類）

`scripts/run_freeze_validloss_ci_9b.py`（`gpu_free_memory_deferred` の直後）に純 helper を追加:

```python
_OOM_EXCEPTION_TYPES = tuple(
    t for t in (
        getattr(torch, "OutOfMemoryError", None),
        getattr(torch.cuda, "OutOfMemoryError", None),
        getattr(torch, "AcceleratorError", None),
    )
    if isinstance(t, type)
)

def is_cuda_oom(exc: BaseException) -> bool:
    """True iff exc is a CUDA out-of-memory error (retriable, not fatal).

    torch 2.12 surfaces device-dispatch OOM as ``AcceleratorError`` — a
    *sibling* of ``OutOfMemoryError`` (both are ``RuntimeError`` subclasses;
    neither parents the other), so a pure ``isinstance(OutOfMemoryError)``
    check misses the device-dispatch path. Match by type *and* message:
    """
    if _OOM_EXCEPTION_TYPES and isinstance(exc, _OOM_EXCEPTION_TYPES):
        return "out of memory" in str(exc).lower()
    if isinstance(exc, RuntimeError):
        return "out of memory" in str(exc).lower()
    return False
```

`main()` の `except OutputPathDiedDuringRun` handler の直後に OOM handler を追加:

```python
except RuntimeError as exc:
    if is_cuda_oom(exc):
        logger.error(
            "CUDA out-of-memory during the run (%s). A concurrent process "
            "is likely holding the card ... Deferring — exit %d (EX_TEMPFAIL, "
            "retry later); completed arms stay banked in the --ledger.",
            type(exc).__name__, EXIT_GPU_TEMPFAIL,
        )
        return EXIT_GPU_TEMPFAIL
    raise
```

exit-code contract docstring（line ~156）と `launch_freeze_ci_9b_full.py` docstring（line 28）を更新: `75 GPU free-memory tempfail OR run-time CUDA OOM (contention) → RETRY`。

**設計理由**: tempfail 化により launcher は 120s backoff → retry で GPU 窗口を poll し続け（`--ledger` が完了 arm を bank するので進捗は永続）・窗口が開けば 9 arms × 1500 step を run する。fatal のままでは task が死に、operator が手動再発火せねばならなかった（feedback lever #1「unblock/launch」の真の障害）。

## 受け入れ test（`tests/test_run_freeze_validloss_ci_9b.py`・`TestCudaOomTempfail` +8・GPU-free・mutation 証明）

- **`is_cuda_oom` の typed-match（3 parametrized）**: `torch.cuda.OutOfMemoryError("out of memory")` / `RuntimeError("CUDA error: out of memory")` / `RuntimeError("out of memory allocating ...")` が全て True（torch 2.12 の `AcceleratorError` が sibling である事実を踏まえ、typed-match + message-match の両方で捕る）。
- **`is_cuda_oom` の non-OOM reject（3 parametrized）**: `RuntimeError("device-side assert")` / `RuntimeError("connection broken")` / `ValueError("out of memory")`（RuntimeError だが OOM でない）/ `KeyboardInterrupt` が False（OOM でない RuntimeError は fatal のまま reraise）。
- **`test_main_classifies_run_time_oom_as_tempfail`**: `--min-free-gib 0`（pre-flight bypass）+ `monkeypatch.setattr(mod, "run_ci_9b", _boom)`（`_boom` が `AcceleratorError("out of memory")` を raise）→ `main(argv)` が `EXIT_GPU_TEMPFAIL` を返す。
- **`test_main_reraises_non_oom_runtime_error`**: 同 setup で `_boom` が non-OOM `RuntimeError` を raise → `main` が **reraise**（exit 75 で吞み込まない = fatal は fatal のまま）。
- **`test_fire_script_robustly_launches_full_budget_heterogeneous`**（`TestFullBudgetHeterogeneousLaunchPath` に追加）: committed `fire_freeze_ci_9b_full_heterogeneous.sh` が実行可能・`--n-control 3`（direction-isolation A/B）・distinct deposit/ledger・`for flag in (--config,--ledger,--output): assert f'{flag} "$STABLE/' in text`（全 path が `$STABLE` に anchor = worktree 削除で永続化不能にならない）。
