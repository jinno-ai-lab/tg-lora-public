## 背景（schema guard は missing-FIELD のみ・row-0 のみを閉じ、empty-VALUE を残す）

`scripts/run_freeze_validloss_ci_9b.py::_load_dolly_records` は `7c4aebf` で `_assert_dolly_schema` を**最初の行のみ**（`if i == 0:`）呼び、**key の presence**（`k not in row`）のみを検査する。これは「非 Dolly dataset が `.get(...,"")` で空レコードに化ける」腐敗経路を閉じた。

しかし schema guard の**自身の docstring** が記述する腐敗機構は、missing-FIELD だけでなく**empty-VALUE** でも同じように発火する：

- `response=""`（key present・値 empty）→ `build_sft_example` の `completion = "{response}<|im_end|>"` が bare `"<|im_end|>"` になる
- この終端 tail は mask された prompt prefix の**後ろ**に位置する → labels は all-`-100` で**ない** → `build_real_batches` はこの example を**drop せず保持**する
- arm は junk example で学習しつつ**有効な `valid_loss` を出力**する → corrupt-but-green §4 verdict（GOAL §7・schema guard docstring の**記述通り**の hazard）

schema guard はこれを**捉えない**：(1) presence のみ（key は存在する）、(2) row-0 のみ（後続行の empty response は未検査）。ゆえに private-`src.data` drop-in path に「row 0 は正常だが後続行に empty response を含む dataset」が流れた場合、silent に corrupt-but-green になる。

到達性: `_load_dolly_records` は消費した**全行**を返し `build_real_batches` に渡す。empty-response レコード 1 件でも消費 slice にあれば degenerate batch が生成される。real Dolly では instruction/response は常に非空だが、drop-in dataset では保証されない（`7c4aebf` が「private-`src.data` drop-in path の latent gap」と位置づけたのと同一 context）。

## 設計判断（per-row content guard・schema guard との責務分離・`.strip()`）

- **helper 位置**: `_assert_dolly_content_non_empty(dataset, index, instruction, response)` を `_assert_dolly_schema` の直後に新設。schema guard は「missing-FIELD 扉（row 0）」・本 helper は「empty-VALUE 扉（全消費行）」と**扉ごとに 1 guard** で責務分離（`_assert_*` 命名規約に合致）。
- **検査対象**: `instruction` と `response` のみ（`str(value).strip()` が空か）。`context` は real Dolly で legitimately optional なので**検査しない**（空 context は正常）。
- **`.strip()` 採用**: 真偽値判定でなく `.strip()` — whitespace-only response（`"  \n  "`）は truthy だが bare terminator tail と同一 hazard。mechanism test `test_whitespace_response_is_also_non_skipped` がこれを実証。
- **per-row 適用**: schema guard の row-0 制限を継承せず、消費した**全行**に適用。後続行の key 欠落も `.get(...,"")` → empty としてここで捕捉される（schema guard が row-0 のみ見る per-row missing-key 扉も**結果的に閉じる**）。
- **real Dolly で byte-identical**: real Dolly は常に非空 instruction/response なので本 guard は発火しない → committed verdict/fixture は不変（`test_load_dolly_records_content_guard_byte_identical_for_real_dolly` が実証）。

## fix

`scripts/run_freeze_validloss_ci_9b.py`:

1. `_assert_dolly_content_non_empty(dataset, index, instruction, response)` helper 新設 — `instruction`/`response` の `str(value).strip()` が空のものを列挙し、空があれば `ValueError`（dataset 名・行 index・空 field 名を含む）。docstring に腐敗機構（bare `<|im_end|>` tail が監督付きで drop されない）と `7c4aebf` との同 class 関係（missing-FIELD 扉 vs empty-VALUE 扉）を明記。
2. `_load_dolly_records` の loop body で `instruction`/`context`/`response` をローカル変数に抽出し、`_assert_dolly_content_non_empty(dataset, i, instruction, response)` を**全消費行**に呼んでから append。docstring も schema guard と content guard の二重防衛を明記。

保証: **present-but-empty 値のレコードは schema 通過後でも必ず abort する**。real Dolly corpus では instruction/response は非空なので本 guard は発火せず、committed verdict/fixture は byte-identical。
