## 背景（なぜ「verdict run は非実行可能」は誤りか）

feedback は「robust path で 9B run を起動し §4 TIES を報告せよ」と反復指名する。TASK-0195/0196 はこれに**「9B run は建築的に非実行可能（`train_tg_lora` が L16 で `src.data` を import して即死）」**と応答し、`run_executable_here=false` → SHIP を導出した。本 iter の実測で**これが 2 導線の混同による事実誤り**と判明:

1. **§4 verdict run の entry は worker であり `train_tg_lora` でない**: Makefile `freeze-validloss-ci-9b-full` は `-m scripts.run_freeze_validloss_ci_9b` を直接起動（`train_tg_lora` 経由ではない）。worker は `DEFAULT_DATASET = "databricks/databricks-dolly-15k"`（public Dolly）を `from datasets import load_dataset` で消費し、**`src.data` を一切 import しない**（`grep` で comment のみ・`from src.data`/`import src.data` は 0 件）。
2. **worker は import 成功し `--help` も EXIT 0**: `python -c "import scripts.run_freeze_validloss_ci_9b"` → OK。worker の src.* 依存（`src.model`/`src.tg_lora`/`src.training.config_schema`/`src.utils`）は全て public mirror に存在。
3. **verdict run は既に起動済み（実行可能性の証拠）**: commit 済み両 TIES deposit は**この worker を実際に起動して生成された**（[[dead-cwd-background-run-trap]] の stable-dir fire + harvest）。非実行可能な run が deposit を生成することは不可能。
4. **`train_tg_lora` の `src.data` import は recover.py `--rerun` 導線**: `_build_rerun_command` が `-m src.training.train_tg_lora` を組み立てる（recover.py L512）。ゆえに `src.data` block は **recover 導線**にのみ適用され、**verdict run には適用されない**。

ゆえに TASK-0195/0196 の「9B run は建築的に非実行可能」は verdict run について**誤り**。決定アーティファクト（feedback 命名 deliverable）の中心的主張の事実誤りであり、これを是正する。これは「表面化する、黙殺しない（§7）」の core に直結する（[[ai-hub-feedback-infra-vs-this-repo]] の「false gap を diagnose し直すな」原則の逆＝**実測で真の gap を発見した**場合は是正が正当）。

## Decision trace

- **なぜ verdict worker 由来か（train_tg_lora でなく）**: 決定は §4 verdict について（ship/accept-null/pivot the TIES）。verdict を生成するのは worker（public Dolly）であり、`train_tg_lora`（recover 導線）ではない。決定の `run_executable_here` は verdict run の実行可能性を反映すべき = worker 由来。
- **なぜ推奨 OUTCOME（SHIP）は変わらないのに是正するか**: SHIP 自体は不変（両 verdict TIES 完了）。しかし**理由**が誤り（「run 非実行可能」→正しくは「verdict 完了済み」）。アーティファクトは「machine-verifiable snapshot」を謳うため、中心的事実（実行可能性）の誤りは許容できない。また feedback が「9B 起動」を反復指名する一因が、アーティファクトの「非実行可能」表明（＝block のように読める）にある可能性 → 是正で「既に起動済み・完了」と正確に伝える。
- **選択肢の排除**: src.data 移植は却下（公開境界破壊・[[ai-hub-feedback-infra-vs-this-repo]]）。derived-float guard 延長は TASK-0189 EXHAUSTED → 却下。cache/eval hardening は feedback「STOP」→ 却下。新規 metric guard は feedback 禁止 → 却下。**残った唯一の真の gap は、命名 deliverable 自体の事実誤り（2導線混同）**＝実測で発見。
- **operator 決定への影響**: `run_executable_here` false→true は、operator/feedback に「verdict run は実行可能（既に起動済み）」と正確に伝える。推奨は SHIP 不変（verdict 完了）。PIVOT（絶対 loss・private baseline 比較）は引き続き private-repo action（src.data quality filter 必要）。
