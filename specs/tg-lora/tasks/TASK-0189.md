## 背景（remediation は是正再実行コマンドを印字するだけで起動しない・abort 検出が「報告」止まり）

`scripts/recover.py` は TASK-0063 の runbook を実行可能にする fault recovery CLI である。`apply_remediation` は fault を検出し、OOM checkpoint を sanitize し、低フットプリント recovery config を生成した後、次の結果を append して終了していた：

```python
RecoveryResult("remediate", "ok",
    f"Remediation complete. Resume with: python -m src.training.train_tg_lora "
    f"--config {recovery_config_path} --resume {run_dir}/training_state.pt", ...)
```

これは **operator に再実行コマンドを提示するだけで、再実行を起動しない**。9B arm が step 472 で OOM-killer/SIGKILL により kill された事象（recover.py コメントが system RAM 回収を kill 原因と明記）に対し、operator は手動で copy/paste しなければ完走（step 1500）に至らない。本 TASK はこの「報告止まり」gap を閉鎖し、abort 検出を是正再実行の**自動起動**へ接続する（feedback #4(b)）。

## 仕様

1. **opt-in の是正再実行起動**（安全性）：`apply_remediation(run_dir, config_path, *, rerun=False, launcher=None)`。`rerun=True` かつ recovery config 生成成功時のみ、`_build_rerun_command` が build したコマンドを `launcher`（既定 `_default_rerun_launcher` = `subprocess.run(cmd).returncode`）で起動。**既定 `rerun=False` で挙動完全不変**（plain `--remediate` は従来通り診断+config 生成のみ）。
2. **是正再実行は低フットプリント recovery config を使用**：`_build_rerun_command(recovery_config, run_dir, python=sys.executable)` は `[python, "-m", "src.training.train_tg_lora", "--config", recovery_config, "--resume", <run_dir>/training_state.pt]` を返す。recovery config は既に `data.max_seq_len` 半減 + `training.gradient_checkpointing=True` を適用済み（CUDA fault では batch 半減・accum 倍化）なので、OOM/SIGKILL で kill された元 run を再トリガせず resume する = feedback #4(a) と #4(b) を同時に充足。
3. **注入可能 launcher seam**（検証可能性）：`launcher=None` 既定は `_default_rerun_launcher`（実際の subprocess 起動）。test は recorder を注入し、GPU / private `src.data` なしで起動コマンド・低フットプリント config・opt-in 挙動を検証する。
4. **exit code を RecoveryResult に記録**：`rc==0`→`status="ok"`（`Corrective re-run launched (exit 0)`）・`rc!=0`→`status="warn"`（`Corrective re-run launched (exit {rc})`）・起動例外→`status="error"`。いずれも `details` に `rerun_command`/`returncode`/`fault_type` を保持。
5. **CLI `--rerun`**：`--remediate` と併用。`action="store_true"`・既定 off。
