## 主要実装項目

### 1. 混在障害パターンサイクルテスト 🔵

**信頼性**: 🔵 *train_tg_lora.pyのサイクルループ構造・REQ-044より*

正常受理（eval改善）→ NaN検出（非有限パラメータ）→ 損失悪化拒否（eval悪化）→ 正常受理の4サイクルシナリオで、各サイクル終了時のCycleStateとRandomWalkControllerの状態を検証。

**実装ファイル**: `tests/test_extrapolation_safety_integration.py`（拡張）

### 2. NaN→正常遷移の復旧テスト 🔵

**信頼性**: 🔵 *random_walk_controller.pyのreward/penalize交互呼出パターンより*

NaN検出でalpha減少後、正常受理でalpha増加に復旧することを確認。NaN検出でlr半減後、連続受理でlrがlr_maxに向けて上昇することを確認。

### 3. 障害タイプ混在での状態追跡検証 🔵

**信頼性**: 🔵 *cycle_state.pyのrecord_cycle実装より*

NaN検出サイクルはaccepted=False、損失悪化サイクルもaccepted=False、正常サイクルはaccepted=Trueとして記録され、accepted_count/rejected_countの集計が正しいことを確認。

### 4. rollback履歴管理の検証 🟡

**信頼性**: 🟡 *rollback_manager.pyのpush/pop実装から妥当な推測 — NaN検出パスでのpop呼出パターンはtrain_tg_lora.pyのtry/finally構造に依存*

NaN検出サイクルでもrollback_mgr.save()→rollback()→pop()のライフサイクルが正しく管理され、履歴スタックがリークしないことを確認。

## 基本的なテスト要件

### テストケース1: 4パターン混在サイクル 🔵

**信頼性**: 🔵 *train_tg_lora.pyサイクル構造より*

- cycle 1: 正常受理（eval_loss改善）
- cycle 2: NaN検出（apply_extrapolation後にNaN注入）
- cycle 3: 損失悪化拒否（eval_loss悪化）
- cycle 4: 正常受理（eval_loss改善）
- 結果: accepted_count=2, rejected_count=2

### テストケース2: NaN→正常のalpha復旧 🔵

**信頼性**: 🔵 *random_walk_controller.py reward/penalizeより*

- NaN検出でpenalize → alpha減少
- その後の正常受理でreward → alpha増加
- alphaがNaN検出前の水準に向かって回復

### テストケース3: rollback履歴リーク検証 🔵

**信頼性**: 🔵 *rollback_manager.pyのスタック構造より*

- 4サイクル実行後、rollback_mgrの内部履歴長が期待値（save→pop毎に±1）に一致
- 履歴が無限増大していないことを確認
