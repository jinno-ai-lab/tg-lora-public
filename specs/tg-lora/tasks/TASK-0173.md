## 背景（replay は stamped `evidence_hash` を信頼するだけで再導出しない）

`scripts/replay_freeze_validloss_ci.py` は任意の committed deposit を GPU-free で再判定する chokepoint である。producer（`scripts/run_freeze_validloss_ci_9b.py`）は harvest 時に `evidence_hash` を deposit に stamp する — これは deposit の **EVIDENCE bytes**（raw measurements + run-determining config・決して derived verdict/gate/regime label ではない）の SHA-256 であり、coordinated repaint（floats + CI bounds + verdict label + per-arm provenance を同時に編集して全 derived check を通す）や偶発的な byte drift を可視化する意図。しかし replay gate は:

- `replay_samples` が losses から verdict を再導出し `faithful` で照合 ✓
- `citation_label_stale`/`direction_verdict_stale`/`baseline_verdict_stale`/`negative_control_flagged`/`full_context_effective` が derived label を artifact から再導出 ✓
- `_ledger_losses_stale`/`_ledger_witness_stale` が per-arm losses を外部 ledger に bind（`47226b9`）✓

の一方で、**stamped `evidence_hash` を再導出して照合する検査は一つもなかった**（producer の `_evidence_hash` 定義は torch 依存の producer module 内にあり replay は import できず、replay は stamp を単に trusted string として扱っていた）。

ゆえに**内部的に完全に整合した** hand-edited deposit:

- recorded `verdict` は編集後 losses と一致（`replay_samples` が再算出 → `faithful:True`）
- 全 citation/sub-verdict gate が green
- ledger を持たない deposit（baseline/direction/generalization/heterogeneous_generalization/surrogate の 5 deposit — `47226b9` の ledger binding が到達できない）では ledger check も skip

が、自身の evidence bytes（例: freeze order `candidate_order`・run config `total_steps`・per-arm provenance）を編集されても、stale field も note も**存在せず完全 silent** に通る。これは corrupt-but-green §4 verdict を成立させる silent contract-break（GOAL §7）。proof-of-need（本 TASK の test で実証）: ledger-less の surrogate deposit（記録 SURPASSES）で `candidate_order` を反転すると、verdict は faithful のままで ledger check は全て clean、** stamped `evidence_hash` のみが stale を検出**する。

## 設計（single source of truth leaf → producer stamp + replay cross-check）

`freeze_verdict_honesty.py` と同じ torch-free leaf pattern に従う:

1. **新 leaf** `src/tg_lora/freeze_evidence_hash.py`（torch-free）:
   - `EVIDENCE_HASH_KEYS`（27 key の固定順序 subset — candidate/surrogate/control/baseline losses・freeze order・per-arm provenance・run-determining config・seed）
   - `evidence_hash(deposit)` — key subset を `json.dumps(sort_keys=True, separators=(",",":"))` で正規化し SHA-256 hex を返す。`evidence_hash` 自身は key に含まれない（stamp は冪等）

2. **producer refactor** `scripts/run_freeze_validloss_ci_9b.py`:
   - 重複していた `EVIDENCE_HASH_KEYS` 定義 + `def _evidence_hash` を leaf からの import に置換（`from src.tg_lora.freeze_evidence_hash import (...)`）
   - test back-compat のため `EVIDENCE_HASH_KEYS as EVIDENCE_HASH_KEYS` / `evidence_hash as _evidence_hash` を re-export（`tests/test_run_freeze_validloss_ci_9b.py` の 32 個の evidence/hash test は無修正で green）

3. **replay gate 拡張** `scripts/replay_freeze_validloss_ci.py`:
   - leaf から `evidence_hash` を import（torch 依存なし）
   - `_evidence_hash_stale(data)` — stamp が無ければ `False`（proxy/synthetic/legacy recording の artifact-when-present-else-skip 原則・ledger gate と同一）。有れば `evidence_hash(data) != stored`
   - `EVIDENCE_HASH_STALE` prose note block（`_ledger_witness_stale` と同様に helper gate）
   - `replay_to_json` の machine field `evidence_hash_stale`

**両 call site（prose + field）は同一 helper `_evidence_hash_stale(data)` を経由**し、独立再導出しない — 人間の prose と machine の flag は drift できない（GOAL §6.2）。
