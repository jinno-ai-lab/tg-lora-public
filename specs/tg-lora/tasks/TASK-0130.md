## タスク概要

AI_HUB_MAKE_RUN_FEEDBACKで指摘された最重要項目。Progressive Freezing機能群（activation-matching local loss・freeze schedule・freeze frontier）は各モジュール単体テストで検証済みだが、**実モデルを forward+backward で駆動し、3モジュールを統合したE2Eテストが存在しなかった**。特に「部分凍結多層状態での勾配フロー・schedule/loss 相互作用の回帰」は、モジュール単体テストでは検出できない最高リスクの故障モードである。本タスクでは、LoRAパスがforward graph上に乗る忠実な極小トランスフォーマを駆動し、schedule→controller→frontier→loss の完全な構成を検証する。

## 主要実装項目

### TestScheduleControllerFrontierEndToEnd: 構成の構造的整合 🔵

**信頼性**: 🔵 *freeze_schedule.py / freeze_frontier.py / progressive_freeze.py 実装より*

スケジュール予測 ↔ 実モデル凍結 ↔ frontier 報告の三者一致を、エポックごとに検証:
- `test_loop_freezes_exactly_the_scheduled_layers_each_epoch`: 各エポックで controller が凍結したレイヤ集合が schedule 予測と完全一致
- `test_frontier_realized_depth_matches_actually_frozen_count`: frontier 点の depth が実際の凍結数と一致
- `test_frontier_reduction_is_monotonic_and_origin_anchored`: 単調性 + depth-0 原点の非節約

### TestGradientFlowAcrossPartialFreeze: 部分凍結での勾配パーティション 🔵

**信頼性**: 🔵 *progressive_freeze.py compute_local_loss・apply_freeze 実装より*

最高リスクの回帰（サフィックスカットの層ごとの保持）を検証:
- 2層（5,4）のみ凍結した部分凍結状態で `compute_local_loss` の backward を実行
- フロント層（0-3）の LoRA adapter にのみ非ゼロ勾配、凍結サフィックス（4,5）には勾配なし

### TestLocalLossTrainsFront: 実forwardによる学習信号 🔵

**信頼性**: 🔵 *activation_matching.py・progressive_freeze.py compute_local_loss 実装より*

実際の optimizer step で local loss が減少することを検証（design §3.3: front-vs-xin gap は本物の信号）。

### 忠実なLoRA-in-forward フィクスチャ 🔵

**信頼性**: 🔵 *PEFT LoRA構造より*

既存の proj-only 単体フィクスチャとは異なり、LoRAパスをforward graph上に配置し base を凍結。これにより `apply_freeze` の `requires_grad=False` が**実際に**そのレイヤの勾配フローを遮断する関係を観測可能にする。

**実装ファイル**: `tests/test_progressive_freeze_e2e.py`
