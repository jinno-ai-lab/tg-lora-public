## 背景（`numpy.percentile` が NaN を黙って伝播し verdict を TIES に固定する）

`src/tg_lora/freeze_surrogate_ci.py::surrogate_valid_loss_ci` は candidate / surrogate 両 arm の valid_loss 配列を受け取り、`bootstrap_difference_ci` で `mean(surrogate) − mean(candidate)` の percentile bootstrap CI を計算し、CI が零を含むかで SURPASSES / UNDERSHOOTS / TIES を決定する GOAL §4 唯一の verdict 計算 chokepoint である。proxy `run_ci`（`scripts/run_freeze_validloss_ci.py`）・9B `run_ci_9b`（`scripts/run_freeze_validloss_ci_9b.py`）・committed-deposit replay（`scripts/replay_freeze_validloss_ci.py`）の**全 3 producer 経路**がこの関数に流入する。

`numpy.percentile` は NaN を含む配列上で **NaN を返す**（線形補間が NaN に触れるため）。かつ `nan > 0.0` / `nan < 0.0` はともに `False` である。ゆえに candidate / surrogate 配列の**いずれか一つでも非有限値を含む**と:

- `improvements = surrogate[idx].mean(1) − candidate[idx].mean(1)` が NaN を多数含み（NaN 要素を引く draw の平均は NaN）
- `lower` / `upper` = `percentile(improvements, ...)` = **NaN**
- verdict 分岐 `lower > 0 ⇒ SURPASSES / upper < 0 ⇒ UNDERSHOOTS / else TIES` は `nan > 0`・`nan < 0` とも `False` → **常に TIES**

これは**両 citable full-budget §4 verdict（homogeneous `4b88ca8`・heterogeneous `1224057`）と同じ label** である。ゆえに diverged arm（gradient explosion で NaN/inf valid_loss）のサンプルが bootstrap に流入すると、failure が**真の null と区別できない corrupt-but-green TIES verdict** になる（GOAL §7 silent contract-break）。実証（numpy 2.4.6）: candidate `[1.5, nan, 1.6]` → point/lower/upper 全て `nan` → verdict `TIES`。

到達性: `756ea96` は committed-deposit **形成**（`extract_best_valid_loss`）のみを閉じた。しかし proxy `run_ci` と 9B `run_ci_9b` は live training arm から**形成 chokepoint を経ずに直接** `surrogate_valid_loss_ci` に流入する（`run_ci` は finiteness check 0・`run_ci_9b` の `math.isfinite` check は regime / cost 軸のみで主 verdict 配列は未 guard・`arm_valid_loss_9b` は `last_loss = float("nan")` で初期化し `eval_loss_9b` が非有限値を返しうる）。replay 経路も `load_samples` は non-empty check のみ。ゆえに 3 経路全てで NaN→silent-TIES が到達可能で、`756ea96` の「形成 chokepoint のみで閉じ、replay 側への第二防衛は置かない」スコープ外（形成を経ない proxy/9B 経路）かつより低階層（verdict 計算そのもの）の chokepoint である。

## 設計判断（計算 chokepoint での単一 fail-loud・filter でなく raise）

- **guard 位置**: `surrogate_valid_loss_ci`（全 3 producer が流入する verdict 計算の唯一の entry）。`bootstrap_difference_ci`（下位 math 層）には置かない — `756ea96` の「同軸への N+1 guard 回避」 discipline と既存の `_require_non_empty` 二重検証を踏まえ、verdict entry 1 箇所で全経路を一括保護する。
- **raise でなく filter**: deposit の各 entry は 1 arm の最終結果（per-step ではない）。非有限 entry = その arm は end-to-end で diverged であり「honest な有限 fallback」は存在しない。silent に drop すると `n_candidate`/`n_surrogate` が変わり CI が変わり verdict が変わる = 第 2 の silent corruption。ゆえに `_require_non_empty` と同じ「bootstrap は誠実に進められない」契約で **loud に `ValueError`**。
- **相補性（非重複）**: `756ea96` は committed-deposit **artifact** に NaN が書き込まれるのを防ぐ。本 guard は **verdict 計算**に NaN が流入するのを防ぐ。異なる chokepoint・異なる artifact 範囲・proxy/9B 経路は `756ea96` が触れない。

## fix

`src/tg_lora/freeze_surrogate_ci.py`:

1. `import math` 追加。
2. `_require_finite_losses(values, arm)` helper 新設 — `float(v)` 後 `math.isfinite` で非有限を検出し、arm 名 + 非 finite 値 + corruption 機構（NaN→percentile→NaN→silent-TIES）+ 対処（diverged arm を drop/re-run）を名指しした `ValueError`。
3. `surrogate_valid_loss_ci` で `_require_non_empty` の直後に candidate / surrogate 両 arm へ適用。
4. module docstring の honesty 節に「Non-finite input honesty」項を追加（`nan` が両方向で `False` となり TIES に落ちる機構を文書化）。
