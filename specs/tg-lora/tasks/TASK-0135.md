## タスク概要

steering は前反復（Phase 67/68/69）を「byte-identical-by-default であり関門判定を一つも動かさない」「まだ存在しない証拠の pluggable な着地点をもう一つ増やすだけでなく、実 runtime outcome を動かす方向へ傾けよ」と評した。具体的に挙げられたのは (a) `resolve_level1_ceiling` / `calibrate_reproduction_bracket` を実際の計測源へ配線して判定を FAIL→PASS で反転させる、(b) CPU-proxy A/B 再生産を走らせ `ReproductionRecord` に N>=3 の実観測を詰めて bracket を非薄にする、(c) CUDA A/B 経路の実装、そして distribution-consistency loss が学習経路で活性（非零係数）であることを before/after 値で確認・報告すること。

本タスクは、これまで定数偽造でのみテストされていた §6.2/§6.3 着地点を、Phase 67 in-vivo suite が既に持つ本物の `_BackwardCounter` A/B 計測（live autograd graph の後方走査回数）へ接続する 2 つの adapter を新設し、着地点を実測で厚くする。本 worktree は `torch+cpu`・`src.data` 欠如・9B モデルなしであり CUDA A/B（item c）は実行不能だが、CPU-proxy A/B（item b）は本物の runtime 計測であり、偽造ではない。

配線の結果得られる本物の runtime outcome は以下の通り。**①（item b）** depth {2,3,4} の trio A/B 3 再生産 + 共通 baseline から、`reproduction_record_from_ab_measurements` が N=3 の実観測（非薄）を運び、`compare_freeze_levels` の比較ヘッドラインが**点推定から再生産件数付き bracket へ**遷移する（監査行に `reproduction_bracket` / `calibrated` / `n=3` が現れる）。各 measured headline は model-free accountant の Level-2 予測と 1e-9 で一致し、測定↔予測ループが閉じる。**②（item a の正直な報告）** 同じ A/B から取った Level-1（freeze-only）3 再生産は**全て ~0** に実現する（層の重み勾配を凍結しても activation gradient は層を走査し続けるため）— 非薄 record でも `observed_reduction≈0` なので ceiling は 0.0 に留まり、Level-1 判定は**正直に FAIL を維持**する。ここを PASS へ反転させるには 9B スケール・grad-ckpt 下での実測が必要であり、本 proxy では偽造せず FAIL のままとする（判定経路そのものは実測で生きて観察可能になった）。**③** distribution-consistency loss は純テンソル試験で before=1.0・dist 項=0.5・after=1.5（=before+dist）に固定され、勾配非零を確認、更に trio の学習経路コール `compute_local_loss` にも thread していることを off-pin 状態で観察した。

これは着地点の byte-identical 保証（record なし → `None`/`0.0` → 出力不変）を保ちつつ、実測値が与えられた時にそれを正直に反映する経路を動かしたものであり、gate の厳格化ではない — steering が「既に厳格な gate を硬化するな」とした制約にも合致する。

## 主要実装項目

### 計測源 adapter（`freeze_cost.py` §6.2/§6.3）🔵

**信頼性**: 🔵 *`resolve_level1_ceiling`/`calibrate_reproduction_bracket` が既に消費する `Level1RealizationRecord`/`ReproductionRecord` と同一 dataclass・同一検証契約 / Phase 67 in-vivo `_BackwardCounter` 計測経路より*

- `level1_realization_record_from_measurements`: 各 `level1_backward` を 1 Level-1 再生産の後方走査総数、`baseline_backward` を対となる full-backprop 数とし、観測削減率 `1 − l1/base` の**中央値**を `observed_reduction`、件数を `num_runs` に載せた record を返す。非正 baseline・空・baseline 超過（凍結は走査を増やさない不変量）を `ValueError` で拒否。
- `reproduction_record_from_ab_measurements`: 各 `reproduction_backward` を 1 Level-2（suffix 切断）再生産の後方走査総数とし、観測 headline `1 − repro/base` を観測とする `ReproductionRecord` を返す（`compare_freeze_levels(reproduction_record=)` が直消費）。同一検証契約。
- いずれも source 文字列（既定 `cpu_proxy_level1` / `cpu_proxy_ab`）を監査可能なまま運び、薄い証拠判定は record 自体の `MIN_SAMPLE_FOR_CONFIDENCE_BAND`（=3）に委ねる。着地点側の byte-identical 保証（record なし → 既定）は無傷。

### 実 CPU-proxy A/B 配線（`test_progressive_freeze_invivo.py`）🔵

**信頼性**: 🔵 *Phase 67 suite の `_run_baseline`/`_run_trio`/`_run_level1` + `_BackwardCounter` は既存の本物の live-graph 計測 / accountant 予測は均一コスト第一階モデルで自明より*

- `_accountant_at_depth`（depth 別の凍結スケジュール over 均一 per-layer cost）と module fixture `_ab_repro`（`TOTAL=20`・`WARMUP=5`・depths=(2,3,4) で baseline 1 + trio 3 + level1 3 を実走行）を新設。
- bracket は 3 depth の実 headline（相互に異なる）から非薄の実幅を得る。各 headline は accountant 予測と一致 → 計測が予測を裏切らない（測定↔予測ループの閉包）。

### Level-1 正直 ~0 の判定経路（design §6.2）🟡→🔵

**信頼性**: 🟡 *判定反転（item a の FAIL→PASS）は実プロキシデータでは起きない — 凍結のみでは activation gradient が層を走査し続け削減が ~0 になるという live-graph の実測事実に基づく* → 🔵 *同じ事実が `RealizedReduction` docstring・§6.2 設計に明記済み*

- Level-1 3 再生産 record は非薄だが `observed_reduction≈0`。`resolve_level1_ceiling` は非薄 record でも観測値（0.0）を返し、ceiling は 0.0、`level1.verdict` は **FAIL を維持**。判定経路は実測で駆動され観察可能になったが、正直な測定値ゆえ反転しない。FAIL→PASS 反転の*機構*自体は `test_level1_non_thin_record_resolves_to_observed`（非薄かつ非零の record → ceiling は観測値へ解決）で検証済み。実測での反転は 9B/grad-ckpt 環境の別作業。

### distribution-consistency loss の観察（§3.1 Phase 3）🔵

**信頼性**: 🔵 *固定テンソル PREDICTED/TARGET からの解析値（MSE=1.0・dist 項=0.5・before=1.0・after=1.5）・`compute_local_loss` は trio 学習経路の実コールより*

- 純テンソル（`TestDistributionLossActive`, 5 件）: Phase-1（`dist_weight=0`）は純 MSE=1.0、distribution 項は非零=0.5、Phase-3（`dist_weight=1`）は after=1.5=`before+dist`、`dist_weight=2` で 2.0 に scale、arm は非零勾配を吐く。before/after 値は pinned・観察可能（L3 status observed, not assumed）。
- 学習経路（`TestDistributionLossInTrainingPath`, 1 件）: trio の境界凍結 + `xin` cache 後、`compute_local_loss` の `dist_weight=0` vs `>0` を比較。**正直な体制注記**: trio 定常状態では境界 local loss が front を cached `xin` に pin するためマッチング損失は ~0。arm は front が pin を外れた時（非 pin 学習信号・入力分布 shift 等）に信号を運ぶ — off-pin 状態を front 重みの明示ドリフトで駆動し、arm が training-path scalar を引き上げることを確認（テンソル試験の予測通り）。
