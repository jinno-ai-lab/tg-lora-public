## タスク概要

Guard 実験（10_guard_experiment）のコア制御器 `DynamicFreezeController`（`src/tg_lora/dynamic_freeze.py`）は、§3（出力側からの連続凍結）と §4（上流端からの可逆解放）の双方向を駆動する。§3 は in-vivo trio test で凍結順序が pin されていたが、**§4 の可逆解放経路（`decide_unfreeze` / `apply_unfreeze` / checkpoint 復元）は PyTorch 正本で直接カバレッジ 0** であり、トレーナ（`train_tg_lora.py` L1649-1653）が compute→unfreeze→freeze の順で毎 cycle 呼ぶにもかかわらず検証されていなかった。

本タスクは、この未検証経路に潜んでいた **2つの相互作用欠陥** を TDD で顕在化・閉包する:

1. **同サイクル即時再凍結（リリースの no-op 化）**: `compute_r_A` は固定中レイヤに `0.0` を記録する（`dynamic_freeze.py` L91-95）。`decide_unfreeze` が上流端 L_k を解放しても、解放は compute_r_A 実行後（同 cycle）に起こるため L_k の履歴は `0.0` 埋めのままである。直後の `decide_freeze` が `_is_quiet(L_k)=True` と判定し L_k を同サイクルで再凍結する。リリース＋再凍結が相殺され、§4 の「上流から可逆に解放して再学習させる」という設計意図（§1 クレーム）が**全く実現されない**。

2. **stir のドレイン化（周期性の喪失）**: `apply_unfreeze` は `_frozen_since_cycle` を更新しない。stir は「固定塊が R cycle 維持されたら上流端 1 層を解放」と意図されるが、リリース後も `cycles_frozen = cycle - frozen_since` は単調増大し続けるため、以降毎 cycle stir が再発火しブロックを 1層/cycle でドレインする。（欠陥1の即時再凍結が `apply_freeze` 側で frozen_since をリセットするため、結果的にタイマは巻き戻るが、それは「リリースが no-op だったから」であり、欠陥1が直れば本欠陥も表面化する。）

欠陥1は GOAL §7 鉄則の典型例である — 「released L3」と log されるが実態は no-op。中間指標（log/カウンタ）だけで「§4 が効いている」と結論すれば、GPU 計測前に不正確な前提でランを組むことになる。本タスクは GPU ブロック（Category C）に先立ち、CPU で機械的に検証可能なこの正確性欠陥を閉じる。

## 主要実装項目

### `apply_unfreeze` の署名拡張とタイマ再武装・解放記録（`dynamic_freeze.py`）🔵

**信頼性**: 🔵 *`cycle` を keyword-only（default `None`）とし、ad-hoc/test 呼出点の既存契約を保存 / リリース時の `_frozen_since_cycle = cycle` は「固定塊が変化したので R-cycle 保持カウンタをリセット」という §4(a) の設計意図の直接符号化 / `_released_at` 記録と固定期 `0.0` 履歴破棄はクールダウン判定の前提 / 唯一の本呼出点（train_tg_lora）は `cycle=cycle` で更新済み*

- `cycle` 受け取り時のみ副作用（再武装・記録・履歴破棄）を発火。`None` は旧契約（decide/apply ループを回さない ad-hoc 呼出）の挙動保存。タイマ再武装により stir は「1層解放 → 再待機」の周期挙動となり、1層/cycle のドレインを排除。

### `_in_release_cooldown` + `decide_freeze` scan ガード（同ファイル）🔵

**信頼性**: 🔵 *クールダウン長 = `window` は「1 cycle に 1 実測 r_A → 完全な実測 window 蓄積まで再凍結不可」という §3 warmup（`if cycle < window: return []`）と同じ証拠要求原則 / scan は `break`（当該層は block の adjacency point = `min(block)-1` なので通過拡張は非連続で元来拒否される・早抜けの方が意図明確）/ 既存 in-vivo trio test は apply_unfreeze 未使用のため `_released_at` が空 → ガード非発火 → 挙動不変*

- リリース直後の「固定期 0.0 履歴による見せかけの静穏」で §3 が同サイクル再凍結する経路を遮断。満了後は真の `_is_quiet` 判定に戻り、真に静穏化していれば再凍結（block 再成長）する — これが §4「可逆」の実現。

### `DynFreezeState.released_at` 追加と寛容な復元（同ファイル）🔵

**信頼性**: 🔵 *dataclass フィールド `default_factory=dict` で新規構築は安全 / `load_state_dict` は `getattr(state, "released_at", {})` でフィールド欠損旧 checkpoint を破壊なく読込 / テスト `test_load_state_dict_tolerates_legacy_state_without_released_at` で属性欠損オブジェクトからの復元を固定*

- 復元直後にリリースクールダウン中だったレイヤが即時再凍結されるのを防ぐ。研究 [UNVERIFIED] とはいえ、チェックポイント復元の正確性を構造的に保証。
