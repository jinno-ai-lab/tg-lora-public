## 背景（なぜ「run を起動する」ではなく「決定を code 化し PIVOT を訂正する」か）

feedback（value-judge）は「guard/judge/advisor polish を止め、9B run を robust path で起動して §4 TIES を報告せよ」と継続指名し、GPU block の場合は「operator 決定 + unblock step の表面化」を sanctioned fallback とする。検証の結果:

1. **両 citable full §4 verdict は既に landed TIES**（TASK-0194 で GPU-free replay 確証済）→ 再起動は既知 TIES の再現（新情報ゼロ）。
2. **9B run はこの公開 mirror で建築的に非実行可能**: `src.training.train_tg_lora` は module scope（L16）で `from src.data.build_seed_dataset import load_dataset` するが、**`src.data` はこの公開 mirror から意図的に剥奪**されている（`scripts/prepare_data.py` が "the private data pipeline that is stripped from this public mirror" と明記・`tests/test_filter_dataset.py`/`tests/test_dedup.py` は interface だけ残存し実装は不在）。ゆえに GPU にかかわらず起動即 import error。
3. **GPU は一時的要因**（seq1024 full-budget は 12GB 3060 上 suffix-only config で fit・両 leg を同 HW で実施済）→ 恒久ブロッカーは (2) の `src.data` 剥奪。

ゆえに run 起動は成立せず、sanctioned fallback「operator 決定の表面化」を取る。ただし TASK-0194 はそれを **docs-only** で行い、かつ **PIVOT unblock step を「`src/data/filter_dataset.py` 移植（この mirror 内）」と記載**していた。これは上記 (2) の意図的剥奪と矛盾し、PIVOT を誤って public 側選択肢として提示していた。本 TASK は:

- 決定を docs から**実行可能アーティファクト**（`scripts/section4_operator_decision.py` + `make section4-operator-decision`）へ格上げし、feedback loop/operator が「起動したか？」と再問する代わりに**1 コマンドで決定状態を確認**できるようにする。
- PIVOT を正しく **private-repo 専用**（`/home/jinno/tg-lora`）と訂正し、public 側の実行可能決定を SHIP（推奨）/ ACCEPT-NULL に絞る。

## Decision trace

- **run 起動以外の選択肢の排除**: src.data 移植（TASK-0194 PIVOT 記載）は `scripts/prepare_data.py` の "stripped from this public mirror" 注記と矛盾 → 公開 mirror の意図的境界を破るため却下。recover.py への更なる guard は feedback が「収益逓減・STOP adding tooling」と明示 → 却下。derived-float guard 延長は TASK-0189 EXHAUSTED 宣言 → 却下。spine/judge/advisor は feedback 禁止 → 却下。
- **なぜ新規スクリプトか（docs-only でないか）**: TASK-0194 は docs-only 決定表面化を行ったが feedback は「9B run 起動」を継続指名＝docs-only を不十分と判断。feedback が制裁する fallback「決定の表面化」を**実行可能アーティファクト**に格上げすることが、docs-only を超える最小の前進。新規スクリプトは「起動するか再問」を「1 コマンドで確認」に置換し、loop の反復を閉じる設計。
- **なぜ PIVOT 訂正が荷重を担うか**: TASK-0194 の PIVOT「src.data 移植（mirror 内）」は、唯一残る open（DATA 軸）の unblock step として誤って public 側に提示されていた。これが「移植すればこの mirror で発火できる」という偽の希望を生み、feedback の「起動せよ」持続の一因。本訂正は PIVOT を private-repo 専用とし、public 側決定を SHIP/ACCEPT-NULL に確定＝決定空間を**実質的に動かす**（feedback が求める「operator 決定を動かす」に合致・polish でない）。
