## 背景（launcher は worker exit 78 を distinct 分岐で扱わず `unknown` 経路に流れ retry し続ける）

`scripts/launch_freeze_ci_9b_full.py::classify_exit_code` の現状:

```python
# 既存 4 + 1 分岐（簡略化）
if code == 0: return Decision(Action.DONE, "done", 0.0)
if code == 1: return Decision(Action.FATAL, "unexpected_exit", 0.0)
if code == 2: return Decision(Action.FATAL, "cuda_down", 0.0)
if code == 3: return Decision(Action.FATAL, "incomplete_resume", 0.0)
if code == 75: return Decision(Action.RETRY, "gpu_tempfail", backoff)
if code < 0: return Decision(Action.RETRY, "signal_kill", backoff)
return Decision(Action.RETRY, "unknown", backoff)  # ← 78 はここに落ちる
```

TASK-0180/0181 が producer / replay 側で `exit 78`（operator error）を発生させても、launcher 側では `unknown` 経路に流れ `Action.RETRY` 扱いになる = **operator が 1 回目で修正可能な config / YAML / schema 違反を launcher が無限 retry** = silent cost 増大 + `b8ee35c` 5 invariants の deadline 機構と矛盾。

AI_HUB_MAKE_RUN_FEEDBACK は launcher 側について直接言及していないが、operator-facing follow-up の自然な延長として exit code 分類を整合させる必要あり。本 TASK は launcher 側で 78 → FATAL 分岐を追加し、worker exit 78 を **retry 不能 / deadline 強制** として surface する。

## 設計（`classify_exit_code` への 1 分岐追加 + 既存 signal-kill RETRY の手前配置）

- **`classify_exit_code` の修正**: 既存 4 + 1 分岐の **直後・signal-kill RETRY の直前** に以下を追加（signal-kill が `code < 0` で先に catch される 78 と無関係・順序重要ではないが、可読性のため `code == 75` の直後に配置）
  ```python
  if code == 78:  # EXIT_OPERATOR_ERROR = sysexits.h EX_CONFIG
      return Decision(Action.FATAL, "operator_error", 0.0)
  ```
  これで worker が `exit 78` で死ぬと、launcher は **即座に `Action.FATAL` を返して deadline 経由** で process 終了 = operator error の retry による silent cost 増大を止める。
- **既存 logic 無変更**: 4 worker exit code 分岐 + `GPU_TEMPFAIL=75` 分岐 + signal-kill RETRY + `unknown` RETRY default は **byte-identical 維持** = `ad8c84a` 4 worker exit code contract + `b8ee35c` 5 launch-honesty invariants は無変更。
- **新規 5 個目 worker exit code 契約 pin**: `tests/test_worker_launcher_exit_contract.py` に **5 個目の pin** を追加 — worker `exit 78` → `classify_exit_code(78) == Decision(Action.FATAL, "operator_error", 0.0)`。既存 4 個 pin の **追加** であり、既存 4 個は無変更。
- **`LAUNCHER_OPERATOR_ERROR_BRANCH` 定数の import**: `scripts/launch_freeze_ci_9b_full.py` 内で `EXIT_OPERATOR_ERROR = 78` を **module-level constant として再定義** する（launcher は torch-free 設計 + `src.utils.cli_errors` を import する path 依存を避けるため、interfaces.py の `LAUNCHER_OPERATOR_ERROR_BRANCH` の `78` を直接リテラルとして持つ）。`tests/test_worker_launcher_exit_contract.py` 側でも `EXIT_OPERATOR_ERROR = 78` を module-level に持ち、launcher と同じ値を一カ所で pin する（drift 検出用）。

保証:
- **worker `exit 78`** → `Decision(Action.FATAL, "operator_error", 0.0)`（既存 4 worker exit code と区別・5 個目の独立 pin）
- **既存 4 worker exit code**（0/1/2/3）+ `GPU_TEMPFAIL=75` 分岐は **無変更** = `ad8c84a` 契約 pin 維持
